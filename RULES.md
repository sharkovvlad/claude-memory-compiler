# Правила работы агентов с knowledge base

> Эти правила — для агентов, работающих в проекте NOMS. Они обеспечивают преемственность между сессиями. Несоблюдение = другие агенты не найдут вашу работу и наступят на те же грабли.

## Главное правило: всё описательное — в `claude-memory-compiler/`

**Любые архитектурные документы, ADR, дизайн-доки, решения, договорённости с тимлидом, отчёты о trade-off'ах, обзорные тексты — живут ТОЛЬКО в этом репозитории.**

Конкретно:

| Артефакт | Куда | Куда **НЕ** |
|---|---|---|
| ADR (Architecture Decision Record) | `knowledge/concepts/<topic>-adr.md` | ❌ `docs/adr/` в основном NOMS-репо |
| Концепт / архитектура / паттерн | `knowledge/concepts/<topic>.md` | ❌ inline в `*.md` основного репо |
| Daily-журнал сессии | `daily/YYYY-MM-DD.md` (append если файл уже есть) | ❌ только commit-сообщения |
| Handover между агентами | `handover/YYYY-MM-DD_<scope>.md` (для крупных cutover'ов) | ❌ в чате |
| Recipe / how-to для типовой операции | `knowledge/concepts/<recipe-name>.md` | ❌ только в commit-сообщении |
| Live status / migration HEAD | `MEMORY.md` (`/Users/vladislav/.claude/projects/.../memory/`) | ❌ inline в CLAUDE.md |

**В основном NOMS-репо допустимо:**
- Код (`*.py`), миграции (`*.sql`), тесты (`tests/`), конфиг (`*.ini`, `*.toml`, `*.yml`).
- Краткие inline-комментарии в коде (≤ 1 параграф).
- `CLAUDE.md` — operational guide (правила, конвенции, quick-reference), **не журнал**.

**В основном NOMS-репо ЗАПРЕЩЕНО:**
- Папка `docs/` для архитектурных документов.
- ADR-файлы.
- Журнальные `*.md` ("CHANGELOG", "DECISIONS", "HISTORY").
- Длинные дизайн-доки.

## Жёсткое правило: каждая новая статья → запись в `index.md`

`claude-memory-compiler/knowledge/index.md` — это **единственный путь поиска** для других агентов. Без записи там статья **не существует** для команды.

Каждая новая статья в `knowledge/concepts/` обязана:
1. Получить строку в таблице `index.md` (формат как у соседних строк: `\| [[concepts/<file>]] \| 1-2 предложения суть \| daily/YYYY-MM-DD.md \| YYYY-MM-DD \|`).
2. Описание в `index.md` должно быть достаточно специфичным чтобы агент по grep смог найти статью по характерным словам (`grep -l "freeze"`, `grep -l "headless"` и т.п.).

Если меняете существующую статью **и меняется суть** — обновите её строку в index.md тоже (актуальная дата, обновлённое описание).

## Чек-лист закрытия сессии

Перед сообщением «готово» каждый агент обязан:

1. **Daily-журнал**: `daily/YYYY-MM-DD.md` — что изменено, зачем, применено ли. Append если уже существует.
2. **Handover** (для крупных cutover'ов): `handover/YYYY-MM-DD_<scope>.md` — брифинг следующего агента.
3. **KB-урок** (если наткнулся на gotcha или новый паттерн):
   - Сначала `grep -r` по `claude-memory-compiler/knowledge/concepts/` — не описано ли уже.
   - Если есть подходящая статья — append секцию.
   - Если нет — создать новую + **обязательно** запись в `index.md`.
4. **MEMORY.md update** (если изменилось состояние проекта): новая миграция HEAD, завершённый cutover, удалённый workflow, новый флаг. MEMORY читают **все** агенты в начале сессии.
5. **MEMORY-housekeeping:** проверить размер `MEMORY.md`. > 20 KB или > 150 строк → флаг тимлиду «нужен прогон `/anthropic-skills:consolidate-memory`».

## Тест себя перед `git commit`

Спросите: «Если завтра другой агент придёт с похожей задачей и сделает `grep -r 'sticker' claude-memory-compiler/`, найдёт ли он всё что нужно знать?» Если **нет** — нужна статья. Если **да но устаревшее** — нужно обновить.
