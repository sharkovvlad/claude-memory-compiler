# NotebookLM Code Sync (code_to_nlm.py)

Скрипт для инкрементального синка кодовой базы NOMS в блокнот NotebookLM "NOMS Supabase Data". Загружает Python/JS файлы и CLAUDE.md как Markdown-источники, чтобы NotebookLM мог отвечать на вопросы по коду.

## Key Points

- **`/nlm-noms-code` skill:** запускает `code_to_nlm.py --project /NOMS --notebook-id fa75f4de-...` — инкрементальный синк (только изменённые файлы по SHA-256)
- **Manifest desync:** скрипт записывает хэш файла как "успешно загружен" сразу после HTTP upload, не проверяя реальный статус индексирования в NLM — если NLM упал при обработке, манифест остаётся рассинхроненным
- **Открытая проблема:** файлы из `n8n_code/` и `crons/` стабильно показывают красные иконки в NLM (ошибка индексирования) при каждом синке — причина не установлена
- **EXCLUDE_DIRS:** `claude-memory-compiler/` исключён из синка — это инфраструктура памяти Claude, не релевантная для NOMS-бота
- **Заголовки с `code_` префиксом:** Markdown-обёртка каждого файла получает заголовок вида `# code_filename.py` — для уникальности в NLM и предотвращения коллизий имён

## Details

### Архитектура синка

Скрипт оборачивает каждый `.py` / `.js` файл в Markdown с заголовком и загружает как text/markdown источник в NotebookLM через API. Инкрементальность реализована через SHA-256 хэши в локальном манифесте: если хэш не изменился — файл пропускается.

**Проблема манифеста:** после `POST` к NLM API скрипт сразу пишет хэш в манифест и считает загрузку успешной. Реальное индексирование NLM происходит асинхронно после этого — скрипт не опрашивает статус источника. Если NLM возвращает 200 на upload, но потом не может обработать файл (парсинг, размер, спецсимволы) — иконка становится красной, но манифест показывает "ok". При следующем запуске скрипт видит "уже загружен, не изменён" и не перезагружает.

### Красные файлы — известные паттерны

По наблюдениям на 2026-04-14:
- **Стабильно падают:** файлы из `n8n_code/` (JavaScript code nodes для n8n) и часть файлов из `crons/`
- **Одни и те же файлы** при каждом форс-реапплоаде (30/30 проиндексировано, потом часть снова падает)
- **Что не помогло:** увеличение задержки 1.5s → 3s между загрузками, добавление `code_` префикса к заголовкам

**Гипотезы о причине:**
- Unicode escape-последовательности (`\u{XXXX}`) в JS файлах ломают Markdown-парсер NLM
- Нулевые байты или непечатаемые символы в содержимом
- Превышение внутреннего лимита обработки NLM (не задокументирован)

**Задача к решению:**
1. Проверить содержимое проблемных файлов на нестандартные символы
2. Добавить в скрипт проверку статуса источника через NLM API после upload
3. Реализовать повторную загрузку при ошибке индексирования (retry loop)

### COMPILE_AFTER_HOUR изменение

`flush.py` имеет параметр `COMPILE_AFTER_HOUR` — пороговый час, после которого при закрытии сессии автоматически запускается `compile.py`. Изменён с `18` → `8` в обоих проектах (NOMS + PingWin):

- При `18`: компиляция только если сессия закрыта после 18:00 — большинство рабочих сессий остаются не скомпилированы
- При `8`: любая сессия после 8:00 утра триггерит компиляцию знаний — соответствует реальному рабочему ритму

`daily/` логи пишутся всегда независимо от этого параметра — данные не теряются.
