### Lessons

1. **Storage dependency была не synchronized.** Когда сессия началась — storage agent ещё не был spawn'нут. Я подготовил mono-mig с `to_regclass`-guard'ом для audit_log INSERTs (defensive net на случай rollback storage). Параллельный agent сделал storage за вечер (mig 239 merged до моего apply). Pattern: для guard'ов, пишущих в инфраструктурные таблицы — всегда `to_regclass` check, не assumption «таблица будет existed».

2. **Sentinel BEGIN/ROLLBACK с прогревом v7 в той же tx работает.** Можно apply миграцию + сразу запустить sentinels против ТОЛЬКО-ЧТО-applied функции, всё внутри одной транзакции. ROLLBACK откатывает и функцию, и sentinel users — zero side effects. Это сократило диагностический цикл «применить → проверить → откатить» с минут до секунд. Pattern идеален для CREATE OR REPLACE FUNCTION migrations.

3. **«Expected: target=3090» в spec был неверным baseline.** Spec предполагал `phenotype='monw'` (LBM proxy = weight × 0.85), а я в INSERT не задал phenotype → default phenotype используется. Реальный v6=v7=2415 для default phenotype. Не блокер — sentinel'ы #7/#8 предназначены для проверки «zero drift v6→v7», абсолютные числа второстепенны. Lesson: при копировании sentinel-spec из roadmap всегда явно указывать `phenotype`, `pal_coefficient` и все опциональные поля — иначе baseline matches только если spec автор и agent одинаково проинтерпретировали defaults.

4. **`v_user.goal_speed := 'slow'` в plpgsql работает.** Можно мутировать field of a record variable, загруженного через `SELECT * INTO v_user FROM users`. Step 6's `COALESCE(v_user.goal_speed, 'normal')` после этого читает clamp'нутое 'slow'. Без необходимости вводить отдельную переменную `v_effective_goal_speed`.
