## Каскад источников: USDA + GPT-fallback по EAN — PR #364 (mig 488 LIVE)

**Цель:** расширить barcode lookup из одноисточниковой (только OpenFoodFacts) в **4-tier cascade** для weak-coverage regions (RU/IR/IN/AR) и брендов которых нет в OFF.

```
cache → OpenFoodFacts → USDA FoodData Central → GPT-fallback по EAN
```

### Tier 3: USDA FoodData Central (`services/usda_client.py`)

- Free API US Department of Agriculture (signup 30 сек, 1000 req/hour)
- Силён для **американских брендов** (Coca-Cola, Pepsi, Kraft, General Mills) которые широко распространены в LATAM
- Поиск по UPC в Branded foods: `GET /fdc/v1/foods/search?query=<UPC>&dataType=Branded&api_key=...`
- Nutrient IDs: 1008=kcal, 1003=protein, 1004=fat, 1005=carbs (per 100g)
- **Активация:** owner кладёт `USDA_API_KEY=...` в `/home/taskbot/noms/.env` + рестарт `noms-webhooks`. Без env — слой no-ops silently → каскад идёт дальше
- Защита: `servingSizeUnit='ml'` → НЕ выставлять `package_grams` (плотность неизвестна, мл≠г)

### Tier 4: GPT-fallback по EAN

- `services/barcode.py::fetch_gpt_fallback` — gpt-4o-mini structured output
- Промпт-инструкции: «угадывать категорию хуже чем сказать "не знаю"»; confidence floor 0.7; is_known=false → discard
- **EAN-prefix → country mapping** (~85 GS1 ranges в `_EAN_PREFIX_COUNTRY`): префикс 460 = RU, 626 = IR, 890 = IN, 482 = UA, 750 = MX, 789-790 = BR, 84 = ES, ...
- Это feeds промпт как hint: «Barcode 4607097960058 / EAN prefix country: RU / User country hint: RU → what product?»
- Цена: копейки за запрос (~250 токенов roundtrip)
- **Гейт:** флаг `gpt_barcode_fallback_enabled` (default OFF) — переключение через app_constants без рестарта (hot-reload через v_user_context)

### Каскад в `lookup_product`

- Каждый non-cache hit прогревает `barcode_cache` с правильным `source` (`openfoodfacts` / `usda` / `gpt_inferred`) → следующее сканирование = cache hit
- Layer exceptions изолированы — USDA упал → каскад продолжается на GPT
- Все 4 layer'а отвязаны (моки в тестах легко изолируют каждый)

### Wire в `handlers/food_log.py::_try_barcode_lookup`

- Читает 2 флага из `ctx.constants`: `barcode_cascade_use_usda` (default ON), `gpt_barcode_fallback_enabled` (default OFF)
- Lazy OpenAI client init — только когда GPT-fallback включён (не создаём клиент для пользователей где флаг OFF)
- Пробрасывает `lang_code` + `country_code` в каскад

### Durable lessons

1. **Связанные поля = каскад.** Любое поле, осмысленное только при определённом значении другого поля (gender / pregnancy / age), требует либо триггера-каскада на уровне БД, либо CHECK constraint. Доверять «никто никогда не оставит грязное состояние» нельзя — owner это и сделал тестом.
2. **Защита в несколько слоёв.** Каскад чистит данные, gender-гейт в функции страхует от data-drift, RPC `apply_daily_modifier` валидирует ещё раз, Python проверяет `applied=true` перед действием. Любые два уровня могут отказать — третий сработает.
3. **`applied=false` ≠ `error`.** Postgres RPC, возвращающие jsonb, часто пишут `{"applied":false,"suppressed":true,"reason":"..."}` для «мягких» отказов. Питон-проверка только по `error` — ложно-положительный success.
4. **Текст-обещание = действие.** Любое cron-сообщение вида «added +N» требует, чтобы код перед отправкой записал то самое действие. Иначе годами шлёшь ложь и узнаёшь об этом случайно.
5. **Hard-coded цифры в i18n-переводах ломаются при изменении business-логики.** «+175» сидело в 13 языках и не покрывало `luteal_early=100`. Placeholder `{kcal_delta}` решает.
6. **Worktree-навигация:** `cd /Users/vladislav/Documents/NOMS` в bash из worktree → переход в **другой клон** (главный, привязан к другой ветке). `git status` оттуда показывает огромный destructive diff, который не имеет отношения к твоей работе. Работать только с относительными путями от текущего worktree, либо использовать absolute path к своему worktree.
