**Архитектурный совет (тимлид + НЛМ):** не плодить micro-статусы phenotype_q1/q2/...; единый макро-статус FSM `edit_phenotype` + step вывод из JSONB — Stateless Sub-FSM паттерн. Единая кнопка cmd_back, context-aware semantic в SQL FSM (Headless SDUI: клиент остаётся Dumb Renderer). Reuse `pop_nav` helper.

**Mig 197** (utvержден):

```sql
IF v_callback = 'cmd_back' AND v_status = 'edit_phenotype' THEN
    SELECT phenotype_answers INTO v_ans_pheno FROM users WHERE telegram_id=p_telegram_id;
    v_answered := CASE
        WHEN v_ans_pheno ? 'q4' THEN 4
        WHEN v_ans_pheno ? 'q3' THEN 3
        WHEN v_ans_pheno ? 'q2' THEN 2
        WHEN v_ans_pheno ? 'q1' THEN 1
        ELSE 0
    END;
    IF v_answered = 0 THEN
        -- exit + pop_nav
        UPDATE users SET status='registered' WHERE telegram_id=p_telegram_id;
        PERFORM pop_nav(p_telegram_id);
        RETURN render_screen(p_telegram_id, 'my_plan');
    END IF;
    -- rollback last answer + render previous Q
    UPDATE users SET phenotype_answers = v_ans_pheno - ('q'||v_answered)::text WHERE telegram_id=p_telegram_id;
    IF v_answered = 1 THEN
        RETURN render_screen(p_telegram_id, 'edit_phenotype');
    ELSE
        RETURN render_screen(p_telegram_id, 'phenotype_q'||v_answered);
    END IF;
END IF;
```

## Sticker Architecture Foundation (Channel A live)

**Контекст:** 3 разрозненных способа отправки стикеров (thinking-индикатор из proxy, freeze-push из cron, заложенная-но-нерабочая декларация `ui_screens.meta.show_sticker` для welcome/success). Планируется ~30–40 стикеров. Без фундамента — каскад тактических правок.

**Решение:** ADR 0001 — 4 канала с единой таблицей `bot_stickers` как source-of-truth и общим in-memory кэшем. Stage 1 этой сессии — Channel A (UI content stickers, headless).

**Сделано (mig 198 + Python Stage 1):**

- `migrations/198_stickers_unified_foundation.sql` — применена. 4 блока:
  - `bot_stickers` + `meta jsonb` + `updated_at` + `trg_set_updated_at` trigger + partial index `(category, sort_order) WHERE is_active`.
  - `render_screen` Step 8b + extra `sticker_category` поле в `telegram_ui` (точечно над baseline `_baseline_render_screen_2026-05-11.sql`).
  - Seed: `onboarding_welcome_1` активирован, `onboarding_success_1` создан (file_id от владельца).
  - `ui_screens.onboarding_success.meta` ← `show_sticker:true, sticker_category:'onboarding_success'` (через jsonb `||` сохранил `no_back`, `canonical`).
- `services/stickers_cache.py` (284 LOC) — in-memory dict, TTL 60s + background refresher, fail-open keep-stale, exp backoff на retry, sync `lookup()`, DI через `query_fn`.
- `services/template_engine.py` — +23 строки Channel A блок (после ACQ FIRST, перед основным item). Lazy import `services.stickers_cache` ради anti-cycle.
- `webhook_server.py` — startup wires `prewarm()` + `start_background_refresher()`; shutdown stops refresher; новый `POST /internal/stickers/reload` с защитой `X-Stickers-Reload-Token` (env). 503 если token не настроен.

**Race-condition guard:** `services/telegram_send.py` отправляет `send_sticker` строго через `await` (не `asyncio.create_task`). **Coexistence:** Stage 1 НЕ трогает `telegram_proxy.py:_stickers_cache` / `_freeze_sticker_cache`. Старая инфра Channel D работает параллельно с новым cache. Stage 2 (отдельный PR) консолидирует.
