### Bug-finding (root cause UAT-blocker)

`handlers/location.py:415-421` в `_render_onboarding_success` **инжектил** `callback_message_id` в `primary_ui` — это паттерн для `render_strategy='replace_existing'`. Но `onboarding_success.render_strategy='noop'` (mig 208) — для noop template_engine инжект **игнорирует**. Предыдущий агент задокументировал паттерн правильно, но `handlers/location.py` оставался со старой логикой и устаревшим комментарием «render_strategy='send_new' so this will be ignored».

### Архитектурные решения (тимлид 2026-05-13)

При обсуждении плана PR B тимлид остановил initial-предложение перехватывать `cmd_back` в Python:

1. **Headless DB ↔ Python разделение.** Статические кнопки (с зафиксированным `meta` в `ui_screen_buttons`) → SQL FSM. Динамические (`loc_country_<CC>`, `loc_tz_<TZ>`) → Python handler.
2. **`cmd_back` в edit-context — статика**: psycopg2 spot-check показал `ui_screen_buttons.meta = {clear_status: true, target_screen: 'settings'}` для edit_country и edit_timezone. SQL FSM сам обрабатывает через menu_v3 path. Python к этому не лезет.
3. **Setters ≠ FSM**: для финала edit-flow использовать 3 раздельных RPC — `set_user_country`, `set_user_timezone`, `clear_editing_state`. Запрет на `set_user_location(..., 'registered')` в edit-context (последний смешивает data + FSM transition; допустимо для онбординга, не для edit).

### NLM-сюрприз vs handover §3

Handover планировал SQL миграцию (mig 209) для расширения `process_user_input` на edit-flow loc_* callbacks. NLM-консультация выявила: `loc_country_<CC>` / `loc_tz_<TZ>` — динамические (генерируются Python-handler'ом на лету), их **физически нет** в `ui_screen_buttons`. `process_user_input` работает через lookup в этой таблице — он не обрабатывает динамические callbacks.

→ **Phase 6.2 не требует SQL миграции.** Полностью в Python.

### Ключевая lesson для будущих агентов

**`food_logs.id` ≠ `food_logs.meal_id`.** id — PK строки, meal_id — grouping UUID для multi-item meals. Любой RPC работающий с meal-references должен использовать **meal_id**, не id, за исключением точечных lookup'ов конкретной строки. `get_meal_by_id(uuid)` фильтрует по `meal_id`, не `id` — это convention.

Pattern: для editing_meal_id всегда `SELECT meal_id ... ORDER BY consumed_at DESC LIMIT 1`. PR review checkpoint — grep `editing_meal_id` против `SELECT id INTO` в новых миграциях.
