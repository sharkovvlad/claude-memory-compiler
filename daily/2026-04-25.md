#### Bug 2 — кнопки Settings остаются на старом языке после смены

- **Симптом.** После cmd_lang_en на ES-юзере: текст экрана `Idioma: 🇬🇧 English` (новый), но кнопки `País / Zona horaria / Notificaciones / Atrás` остались на испанском.

После save_user_language `users.language_code` обновляется, но trigger.translations устарел → старый язык в render.

Они приходят из view `v_user_context`. Реальный путь: `Get User Context` (Dispatcher) → `Prepare for 04_v3` → `trigger.translations` → `Dumb Renderer`.
- **Fix.** Migration 124 (`migrations/124_process_user_input_lang_refresh.sql`):
  - В RPC после `EXECUTE save_rpc` если `v_save_rpc_name='set_user_language'` → re-`SELECT * INTO v_user` (свежий language_code).
  - В final render блок добавлено `translations_override` (jsonb из `ui_translations.content` для нового языка).
- **Dumb Renderer (n8n) правка:** `const translations = data.translations_override || trigger.translations || {}` — RPC override приоритетнее.
- **Verify ✅** через exec 10547+: после смены ES→EN→UK Settings card сразу на новом языке (текст и все кнопки).

#### Bug 3 — cmd_back из Settings → "👋" вместо Профиля

- **Симптом.** Atrás из Ajustes возвращал welcome-сообщение "👋", а не Profile (вышестоящее меню).
- **Корень.** В **`Route Classifier`** Dispatcher: cmd_back routing был ограничен только статусами edit_*:
  ```js
  if (callback === 'cmd_back' && PROFILE_V5_STATUSES.has(user.status))
  ```
  У registered user (нормальное состояние после save_user_language → status='registered') status не в этом наборе → cmd_back уходил в **legacy `04_Menu`** (sxzbSc61YuHFbV4i), где Back Target Router искал старое имя `profile` (а не `profile_main`) и попадал на дефолтный output → нода `Send Main Menu (Back)` с msg_welcome (👋).

- **Fix.** В `Route Classifier`:

  Покрывает все Headless экраны (Profile, Settings, Stats, Progress, League, Friends, Shop, Quests).

#### Bug 4 — нижняя reply-клавиатура не обновляется после смены языка

- **Симптом.** После save языка: inline_keyboard (кнопки в Settings) на новом языке, но нижняя reply-клавиатура (Mi Día / Progreso / Perfil / Añadir comida) на старом.
- **Корень.** Telegram Bot API: `editMessageText` физически не может обновить reply_keyboard (только inline_keyboard). Reply-keyboard кэшируется клиентом до следующего `sendMessage` с параметром `reply_markup={keyboard:...}`. В headless архитектуре все обычные render'ы идут через `editMessageText`/`sendMessage` с inline_kb → нижняя залипает.
- **Fix.** Migration 124 (тот же файл):
  - RPC при `save_rpc='set_user_language'` устанавливает флаг `v_lang_just_changed`, в response добавляет `reply_keyboard_refresh: true` и `language_code_new`.
  - `Dumb Renderer` (n8n) при флаге prepend'ит дополнительный item: `sendMessage` с текстом `messages.language_changed` (✅ переведён для всех 13 языков) + `reply_markup={keyboard: [[Add food], [My Day, Progress, Profile]], resize_keyboard:true, one_time_keyboard:false}`.
- **Подводный камень в n8n** (потратил 1 итерацию): `out[0].render_strategy = ...` пишет в top-level item, n8n Code-нода падала с "Invalid output format [item N]". Корректный синтаксис: `out[0].json.render_strategy = ...` (внутри обёртки `{json: {...}}`).
- **Подводный камень в порядке сообщений:** изначально Settings = `delete_and_send_new`, Listo = `send_new`. n8n Switch Render Strategy выполняет branch[output_index=1] (delete_and_send_new) **раньше** branch[output_index=2] (send_new) → Settings sendMessage завершался первым → Telegram message_id Settings < Listo → в чате Settings выше Listo. Финальное решение: оба items через `send_new` (одна branch, n8n гарантирует sequential per-item) → Listo первый, Settings второй. Старая Settings card остаётся в истории чата (визуальный clutter, не баг).
- **Verify ✅** exec 10615: 2 sendMessage executions, msg_id 2618 (Settings) и 2619 (Listo). После исправления порядка — Listo получит меньший msg_id.

#### Bug 5 (отложен в roadmap) — "тишина" на первое нажатие Профиль

- **Симптом.** Первый клик "Профіль" / "Profile" / "Perfil" reply-кнопки → бот молчит. Второй (через 2-3 секунды) — отвечает.
- **Корень — race-condition.** В Dispatcher pipeline: `Sync Profile` нода обновляет `users.last_active_at = NOW()` (для metrics/streak). Затем вызывается `process_user_input(...)` → внутри RPC делает `SELECT public.debounce_user_action(p_telegram_id, 1500)` → функция читает `users.last_active_at`, видит что < 1500ms назад, возвращает FALSE → process_user_input возвращает `status='debounced'` → Dumb Renderer отдаёт noop → Telegram получает только Answer Callback Query (или ничего) → пользователь видит молчание.
- **Подтверждение в логах.** exec 10567 (cmd_get_profile): `RPCstatus=debounced screen=None`, DR_items=1 strategy='noop' text=''.
- **Workaround (1 SQL update, 5 минут):** уменьшить cooldown с 1500ms до 500ms — снизит риск, но не уничтожит на медленных connections. Применить:
  ```sql
  -- В process_user_input:
  -- BEGIN v_debounce_ok := public.debounce_user_action(p_telegram_id, 1500);
  -- → debounce_user_action(p_telegram_id, 500);
  ```
- **Архитектурный фикс (отдельная сессия, рекомендую сделать первым в следующей).**
  1. Добавить колонку `users.last_action_ms BIGINT` (epoch ms — для precision).
  2. `debounce_user_action` использует `last_action_ms`, а не `last_active_at`.
  3. RPC обновляет `last_action_ms = (EXTRACT(EPOCH FROM NOW()) * 1000)::bigint` ВНУТРИ debounce_user_action (только при success path).
  4. `Sync Profile` (n8n) продолжает обновлять `last_active_at` как user metric — изолировано от debounce.
  5. Дополнительно — рассмотреть скип debounce для reply-keyboard text inputs (Telegram client всё равно не позволит double-tap reply-кнопки до прихода ответа, в отличие от inline-кнопок где UI не блокируется).

Settings не отправлялся, picker не удалялся. **Урок:** `$('Dumb Renderer').item.json.X` — правильная форма для paired-item access в Code-node → Switch → HTTP node цепочках. `$json` имеет неочевидные quirks при iteration по items.

#### Bug 6 (production hot-fix) — HTTP deleteMessage без continueOnFail блокировал workflow

- **Симптом.** Все reply-keyboard клики (Profile/Progress/My Day) → бот молчит. Множественные ERROR exec в 04_Menu_v3.
- **Корень.** В headless архитектуре `process_user_input` для `p_action_type='text'` (reply-keyboard taps) ставит `render_strategy='delete_and_send_new'` — это паттерн "One Menu" из CLAUDE.md (удалить старое контентное меню + send новое внизу). HTTP deleteMessage пытается удалить `last_bot_message_id`. Если это сообщение **уже удалено** (Telegram cleanup, user manually deleted, bot blocked, любая причина) → API возвращает `400 Bad Request: message to delete not found`.
- **Подводный камень n8n:** `options.ignoreResponseCode: true` НЕ достаточно. Этот flag игнорирует HTTP status code в response, но не **node-level error**. Когда Telegram возвращает 400 + error description, n8n HTTP node всё равно throws → workflow stops → следующая нода (HTTP sendMessage) **не выполняется** → пользователь видит молчание.
- **Fix.** На HTTP deleteMessage установлены top-level флаги: `continueOnFail: true` (legacy) + `onError: 'continueRegularOutput'` (n8n v1.x naming). Теперь при failed delete workflow продолжается через regular output → HTTP sendMessage отправляет новое сообщение.
- **Triggered by**: my one-shot recovery sendMessage (msg_id=2630) был сохранён в `users.last_bot_message_id` через Save Bot Message ноду в 04_Menu_v3. Когда user удалил его вручную, последующие RPC возвращали `last_bot_message_id=2630` для delete, но Telegram уже не помнит этого сообщения → 400 → блокировка.
- **Generalization.** Любая reply-keyboard действие зависит от валидности `last_bot_message_id`. Сбоев много: bot blocked + unblocked, message age >48h, manual delete by user, app restart. Все они должны degrade gracefully — `continueOnFail` is the right defense.



### Session: Phone Collection Strategy — Progressive Profiling (KB capture)

**Что:** Создан KB-документ [[concepts/phone-collection-strategy]] — стратегия сбора телефонных номеров пользователей через `KeyboardButton.request_contact=true`. Перенесён из транзитного плана `/Users/vladislav/.claude/plans/unified-sniffing-treehouse.md` (теперь удалён). Обновлён [[concepts/profile-v5-screens-specs]] в 4 местах для устранения противоречий.

**Зачем:**
- Запрос пользователя оценить потенциал сбора phone после ссылки на [Telegram Login Widget / Native SDK](https://core.telegram.org/bots/telegram-login#native-sdks). Установлено: Login Widget применим только к внешним web/mobile приложениям (OIDC), не к боту. Внутри Telegram phone собирается через reply-keyboard `request_contact`, и только в reply (inline-кнопки не поддерживают).
- Главная ценность phone — **anti-fraud + compliance**, а не AI/Stars/TON (где telegram_id достаточен). Phone критичен для: ambassador payout KYC, regional pricing antifraud, premium trial anti-abuse, account recovery.

**Решение — Progressive Profiling в 3 точках:**

1. **Onboarding шаг 10 (страна)** — **опциональная** reply-кнопка `[📱 Подтвердить регион номером]` рядом с существующей `[📍 Отправить местоположение]`. Юзер волен игнорировать. Anti-fraud для regional pricing (E.164 префикс vs declared country).
2. **Ambassador payout** — **обязательный** новый экран `payout_phone_verify` между Method Picker (12.1) и Wallet Input (12.2). Если phone уже привязан — confirm-режим (показываем последние 4 цифры). Если NULL — request_contact reply-keyboard.
3. **Help screen** — **условная** кнопка `[🔒 Привязать телефон]`, **скрывается** если phone уже привязан. Добровольное recovery.

**Важные технические решения:**

- **🔒 Anti-spoofing обязателен:** Telegram позволяет шарить ЧУЖОЙ контакт из адресной книги. Без проверки `Message.contact.user_id == Message.from.id` злоумышленник привяжет чужой номер. Дублируется в n8n обработчике + RPC параметр `p_contact_user_id`.
- **Dispatcher routing:** новая ветка в Route Classifier для `message.contact` (сейчас обрабатываются только text + callback). См. [[concepts/reply-keyboard-routing-pattern]] как аналог.
- **БД миграция (TBD номер):** `phone_number TEXT`, `phone_verified_at TIMESTAMPTZ`, `phone_source TEXT CHECK (...) IN ('onboarding','payout','settings','quest')` + unique partial index `WHERE phone_number IS NOT NULL`.
- **RPC `set_user_phone(p_telegram_id, p_phone, p_source, p_contact_user_id)`** — E.164 валидация, anti-spoofing, anti-collision (`phone_already_used` exception как fraud signal), `country_phone_mismatch` flag.

## Session: Bug 5 fix (race-condition в Dispatcher) + первый шаг ухода с n8n

**Контекст.** В первой сессии дня (выше) Bug 5 был задиагностирован, но отложен в roadmap. Пользователь подтвердил что баг плавающий и появится снова после любого PUT в Dispatcher. Решено починить полностью — двумя независимыми способами + заложить первый этап миграции 01_Dispatcher на Python.

### Расследование (NLM + grep + GET Dispatcher)

Подтверждено реальным кодом — не предположение:

- **`migrations/043_sync_user_profile.sql`** — `Sync Profile` нода в Dispatcher вызывает `UPDATE users SET last_active_at = now() WHERE telegram_id = ... AND (... OR last_active_at < now() - interval '5 minutes')`.
- **`migrations/061_index_translations_debounce.sql`** — `debounce_user_action` делает `SELECT last_active_at INTO v_last; IF NOW()-v_last < cooldown THEN RETURN FALSE; END IF; UPDATE last_active_at = NOW();`. **SELECT-then-UPDATE без `FOR UPDATE`** — не атомарно.
- **`migrations/084_process_user_input_rpc.sql:272`** — `v_debounce_ok := public.debounce_user_action(p_telegram_id, 1500);`. Это единственный вызов debounce во всём проекте (grep по `debounce_user_action(`).
- **01_Dispatcher.json (GET 2026-04-25 14:28 UTC)** — после `User Exists?` (true-branch) две параллельные ветки: (1) `Sync Profile` (HTTP RPC, **dead-end**, никуда не выходит), (2) `Template Engine → Route Classifier → Prepare for 04_v3 → Execute Workflow → process_user_input → debounce_user_action`. n8n не гарантирует порядок параллельных веток.

**Механика бага:**

| Кто быстрее | Что debounce читает | Результат |
|---|---|---|
| Sync Profile | свежий `T1=NOW()` | `NOW()-T1 < 1500мс` → FALSE → **молчание** |
| process_user_input | старый `T0` | `NOW()-T0 > 1500мс` → TRUE → ответ |

«Через раз» — n8n parallel ordering зависит от latency сети, прогретости пулов, нагрузки на cloud. **Любой PUT в Dispatcher может перетасовать порядок** → баг возвращается.

### Что осталось от Bug 5

**Ничего.** Race-condition устранена двумя независимыми способами:
- БД: атомарный `UPDATE ... WHERE` с условием в фильтре, отдельная колонка от `last_active_at`.
- n8n: Sync Profile удалён, параллельной ветки больше не существует.

Даже если будущий PUT в Dispatcher что-то перетасует — Bug 5 не воспроизведётся, потому что:
- В n8n больше нечем создать race на `last_active_at`.
- А `last_action_ms` пишется только из `debounce_user_action` (single-statement, атомарный UPDATE).

### Cold-start observation (отдельная задача)

Live-test показал что первый клик после простоя ощущается медленнее (estimate +0.5–1.5с к baseline). Источники:
- n8n Cloud worker warmup
- httpx pool TLS handshakes (n8n→Telegram, n8n→Supabase)
- Connection pool n8n→Telegram (новый TCP+TLS)

Это покрывает Блок B плана — Python Outbound Sender. Когда n8n перестанет ходить в Telegram (а это будет делать Python с прогретым httpx-пулом и географически ближе к DC Telegram), cold-start первого клика должен сократиться на 400–1000мс. Отдельной сессией.
