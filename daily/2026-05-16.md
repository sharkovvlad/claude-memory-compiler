### Session (03:17)

**Контекст:** Сравнительный анализ вариантов инфраструктуры n8n (self-host vs полная миграция в Python) и планирование следующих шагов.

**Ключевые обсуждения:**
- Детальный SWOT-анализ двух вариантов: B (self-host n8n на VPS) vs D (полная миграция в Python)
- Замеры реальной latency: cold-start n8n Cloud = 1.3-1.5 сек — главный враг UX
- VPS recon: Hetzner 4 ГБ RAM (3 ГБ свободно), 2 CPU, Docker не установлен, есть чужой `tasktracker-bot` — не трогать
- Объём миграции: ~240 нод в 14 воркфлоу, но 90% — тривиальные httpRequest/code/postgres ноды
- Главный риск Python-миграции — `03_AI_Engine` с 5 langchain.openAi нодами (Vision, Transcribe), но `openai` SDK в Python снимает блокер
- Консультация с «другим ИИ» (NotebookLM) — подтвердил RPC-first архитектуру и грабительские тарифы n8n Cloud

**Принятые решения:**
- **Шаг 0 (немедленно):** self-host n8n Community Edition в Docker на VPS — убирает cold-start и лимит 10K executions бесплатно
- **Шаги 1-7 (эволюционно):** постепенная миграция воркфлоу в Python, каждый поток в свой модуль (`handlers/menu.py`, `handlers/onboarding.py`, `services/ai_engine.py` и т.д.)
- **SQLite** (не Supabase) для внутренней БД n8n — проще, автономнее
- Подписку n8n Cloud отменять только через 3+ дня стабильной работы self-hosted
- Каждый n8n workflow → отдельный Python-модуль (не один монолитный файл)

**Выводы:**
- n8n Cloud cold-start (1.3-1.5 сек) — основной враг latency, не исходящий HTTP к Telegram
- Self-host решает проблему за 1 день без переписывания кода; Python-миграция — правильная цель, но на недели/месяцы
- На VPS живёт чужой проект `tasktracker-bot` — жёсткое ограничение при любых инфраструктурных работах
- Параллельная работа агентов требует координации — brief для следующего агента подготовлен
