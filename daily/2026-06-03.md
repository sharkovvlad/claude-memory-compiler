## «Мой день» показывает фразу про удалённый приём — stale insight cache (mig 440, PR #302, applied LIVE)

**Баг (репро owner'а):** залогировал первый приём → удалил (кнопка) → открыл «Мой день» → фраза Номса всё ещё про удалённую еду, при том что цифры показывают 0 ккал. Нарушение консистентности «число ≠ слова Номса».

**Корень (по коду):** фраза «Мой день» — это **кэш** `users.my_day_insight_*`, чья свежесть (`menu_v3.py:_my_day_cache_fresh`) считается **по времени (≤4ч)+локали, НЕ по содержимому дня**. Путь удаления (`menu_v3.py:_handle_delete_meal_now:926`) чистил кэш **только в in-memory ctx** запроса (`replace(ctx, my_day_insight_at=None)`), а реальную инвалидацию перекладывал на fire-and-forget regen, который (а) гонится со следующим тапом и (б) молча `return False` при любом сбое LLM/JSON (`sage.py:403`). Итог: следующее открытие «Мой день» отдаёт «свежую по времени» устаревшую фразу; regen на этом открытии даже не запускается — кэш «свежий» по 4ч-правилу.

**Решение A (внедрено, mig 440, RPC-only):** инвалидировать кэш **атомарно** в `delete_meal_by_id`:
`WITH deleted AS (DELETE FROM food_logs WHERE meal_id=p_meal_id RETURNING telegram_id) SELECT count(*),max(telegram_id) INTO ...` → `IF v_telegram_id IS NOT NULL THEN UPDATE users SET my_day_insight_at=NULL ...`. NULL-им только timestamp (freshness-гейт падает на `at IS NULL`); stale-текст безвреден (не отдаётся), перезапишется regen'ом. Python-правок не требуется.

**Verify:** NLM (delete_meal_by_id = hard delete из `food_logs`, FK=`telegram_id bigint`, нет `meals`/`user_id`/`deleted_at`) → live `pg_get_functiondef` (база миграции) → applied LIVE psycopg2 → ROLLBACK-тест: insert 2-row meal tid 417002669 + `at=now()` → `delete_meal_by_id` → `deleted_count=2`, rows=0, **`at=NULL`** ✅. p95 (с мака, RTT-доминирован) 181ms; добавленная операция = 1 индексированный UPDATE по PK в том же вызове (не лишний round-trip), server-side ~sub-ms, delete = редкий не-hot путь.

**Durable lesson (→ KB [[concepts/my-day-llm-insight]]):** кэш со свежестью **по времени, а не по содержимому** = баг «stale-content-served-as-fresh». Любая мутация дня (add/edit/delete) обязана инвалидировать кэш; in-memory ctx-инвалидация НЕ персистится в БД; fire-and-forget regen ненадёжен (гонка + silent no-op). Место инвалидации — сама mutation-RPC (атомарно), не Python-хендлер.

**Решение B (бэклог, объяснено owner'у простым языком):** content-aware freshness — штамповать кэш «отпечатком дня» (meals_count + last_meal_at / day-version счётчик), `_my_day_cache_fresh` сверяет отпечаток с текущим состоянием → закрывает ВСЕ пути мутации разом, а не только delete.
