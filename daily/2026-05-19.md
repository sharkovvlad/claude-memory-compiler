**Ключевые обсуждения:**
- Детальный SWOT-анализ двух стратегий: B (self-host n8n на VPS) vs D (полная миграция в Python)
- Реальные замеры: cold-start n8n Cloud = 1.3-1.5 сек — главный враг latency, не исходящий HTTP
- VPS recon: Hetzner 4 ГБ RAM (3 ГБ свободно), 2 CPU, 30 ГБ диска, Docker не установлен
- Объём миграции на Python: ~240 нод в 14 воркфлоу, но 90% — тривиальные (httpRequest/code/postgres → 1-3 строки Python)
- Главный риск Python-миграции — `03_AI_Engine` (5 langchain.openAi нод), но прямой `openai` SDK решает
- На VPS живёт чужой проект `tasktracker-bot` — жёсткое ограничение «не трогать»

**Принятые решения:**
- **Шаг 0 (немедленно):** self-host n8n Community Edition в Docker на VPS — убивает cold-start и лимит 10K executions за 1 день работы
- **Шаги 1-7 (эволюционно):** поэтапная миграция воркфлоу в Python, каждый поток → свой Python-модуль
- **SQLite для внутренней БД n8n** (не Supabase) — проще, меньше зависимостей
- Каждый n8n workflow мигрирует в отдельный Python handler (не монолит)
- n8n Cloud подписку отменять только после 3+ дней стабильной работы self-hosted

**Выводы:**
- n8n Cloud холодный старт (~1.5 сек) — это инфраструктурная проблема, не кодовая; self-host убирает её мгновенно
- Полная миграция в Python — правильная долгосрочная цель, но делать её ради экономии на тарифах = заморозка фич на месяц
- Два подхода не конфликтуют: self-host сейчас → постепенная миграция потом
- Консультация другого ИИ (НЛМ) дала полезный контекст, но не знала что n8n уже self-hosted на VPS и что миграция Python идёт поэтапно

**Key lessons fixed в этой сессии:**

1. **`CREATE OR REPLACE FUNCTION` не может изменить parameter list** — добавление optional p_calc привело к новой overload вместо replace. Решение: `DROP FUNCTION IF EXISTS public.<name>(BIGINT, TEXT)` перед `CREATE OR REPLACE` с новой сигнатурой. Транзакция корректно откатилась при первой попытке; добавил DROP statements и re-apply прошёл. **Канонический pattern для future migrations**: если меняешь сигнатуру (включая добавление optional defaults) — DROP first.

2. **`render_screen` визибилити evaluation = SQL fragment под EXECUTE** (L131-135): `'SELECT (' || cond || ')::boolean FROM public.users u WHERE u.telegram_id=$1'`. Это значит condition может вызывать functions — `has_active_safety_guards(u.telegram_id)` работает. Прецедент в проде — cancel-subscription с `can_cancel`.

3. **render_screen передаёт `text_key` в Python template engine as-is** (L189, L214) — placeholder substitution {count}, {family} в button labels НЕ работает. Substitution есть только для callback_data (L147-153). Architectural pattern для динамики button labels — отсутствует, требует engine extension (отложено в B3).

4. **business_data_rpc вызывается single-arg** (L83: `'SELECT public.' || rpc || '($1)'`) — все RPC с optional defaults поддерживают такой вызов.

### Блок D — n8n SQLITE_READONLY incident

После UPDATE active=0 я сделал `docker cp /tmp/n8n.db noms-n8n:/home/node/.n8n/database.sqlite`. Файл скопировался ownership'ом `root:root` (хост user), а контейнер n8n работает as `node:node` → `SQLITE_READONLY` → crash loop каждые ~30 сек, 1+ час. Eat все workflows: AI Engine (распознавание еды не работает), Location, Onboarding. Python `/telegram/health` отдавал 503 с `n8n:false`.

**Fix:** `docker exec --user root noms-n8n chown node:node /home/node/.n8n/database.sqlite` + restart.

**Lesson:** **никогда не использовать `docker cp` для writable state-файла**. Правильно:
- `docker exec --user <expected> sqlite3` (внутри контейнера, ownership наследуется);
- либо `apt install sqlite3` в контейнере и редактировать там;
- либо `docker cp` + immediate `chown` after.

Создан KB-урок [[concepts/n8n-sqlite-docker-cp-trap]].

## Сессия 5 (поздний вечер) — B3 per-guard click-to-modal (mig 284, PR #125)

**Контекст:** Owner запросил продолжать в той же сессии после merge PR #123. B3 = доставить per-guard click → modal_full UX из mockup Screen 2.

**Architecture decision:** static buttons + visible_condition (Option B из B3 handover), не engine extension. Rationale:
- Zero engine work — использует существующие `visible_condition` (SQL fragment под EXECUTE) + `meta.target_screen` для forward dispatch.
- **Critical empirical finding:** PostgreSQL planner memoизирует `STABLE` function calls в рамках одного `render_screen` запроса. 10× visible_condition evals фактически бесплатные. `render_screen('safety_center')` p50=123ms vs `render_screen('my_plan')` p50=149ms — safety_center даже FASTER baseline на 27ms.

**Что сделано (PR [#125](https://github.com/sharkovvlad/noms-bot/pull/125), mig 284):**
- Helper `_user_active_guards_array(BIGINT) → TEXT[]` (STABLE, PARALLEL SAFE)
- 10 modal screens `gm_<family>_<enum>` (prefix `gm_` instead of `guard_modal_` для удовлетворения `screen_id ~ '^[a-z_][a-z0-9_:]{1,62}$'` constraint — longest enum maternal:maternal_status_unknown_protective_maintain превышал лимит)
- 10 safety_center buttons severity-sorted (hard_block rows 0-3, hard_regulated 4-6, soft_override 7-9), text_key reuses existing banner_title переводы
- 10 back buttons on modal screens
- Bump safety_center back row 0 → 99
- 10 cmd_modal_* в PROFILE_V5_CALLBACKS

**Sentinels — все passed:**
- _user_active_guards_array возвращает корректный список для current user
- 2/10 guard buttons visible (age + maternal pregnancy), 8/10 hidden via visible_condition
- Click cmd_modal_age → target gm_age_..., renders modal_full RU: «Сразу к делу. До 18 жёсткий дефицит без врача — риск...»
- Empty state: 0 guard buttons visible, only back

**Key lesson этой сессии:**

1. **screen_id constraint** (`^[a-z_][a-z0-9_:]{1,62}$`) — max 63 chars total. Длинные enum names требуют коротких префиксов screen_id. Используй `gm_` instead of `guard_modal_` если screen_id под угрозой превысить лимит. Documented в watchlist.

2. **STABLE function memoization works в PostgreSQL planner** — STABLE marker позволяет планировщику кэшировать результат в рамках одного query. Подтверждено эмпирически: 10× visible_condition evals дают **отрицательный** penalty vs simpler render. Знание — позволяет смело использовать function calls в visible_condition при их STABLE-маркировке. **Не accepts вслепую** — bench post-deploy.

3. **Pattern «static buttons + visible_condition» для dynamic content** — clean alternative engine extension когда set of possible items известен и небольшой (≤20). Each item = pre-allocated button с unique visible_condition. Когда engine extension рискованнее (как с render_screen), этот pattern предпочтительнее.
