**Ключевые обсуждения:**
- Детальный SWOT-анализ двух стратегий: B (self-host n8n на VPS) vs D (полная миграция в Python)
- Реальные замеры: cold-start n8n Cloud = 1.3-1.5 сек — главный враг latency, не исходящий HTTP
- VPS recon: Hetzner 4 ГБ RAM (3 ГБ свободно), 2 CPU, 30 ГБ диска, Docker не установлен

- На VPS живёт чужой проект `tasktracker-bot` — жёсткое ограничение «не трогать»

**Принятые решения:**
- **Шаг 0 (немедленно):** self-host n8n Community Edition в Docker на VPS — убивает cold-start и лимит 10K executions за 1 день работы
- **Шаги 1-7 (эволюционно):** поэтапная миграция воркфлоу в Python, каждый поток → свой Python-модуль
- **SQLite для внутренней БД n8n** (не Supabase) — проще, меньше зависимостей
- Каждый n8n workflow мигрирует в отдельный Python handler (не монолит)
- n8n Cloud подписку отменять только после 3+ дней стабильной работы self-hosted

**Выводы:**
- n8n Cloud холодный старт (~1.5 сек) — это инфраструктурная проблема, не кодовая; self-host убирает её мгновенно
- Полная миграция в Python — правильная долгосрочная цель, но делать её ради экономии на тарифах = заморозка фич на месяц
- Два подхода не конфликтуют: self-host сейчас → постепенная миграция потом
- Консультация другого ИИ (НЛМ) дала полезный контекст, но не знала что n8n уже self-hosted на VPS и что миграция Python идёт поэтапно

**Key lessons fixed в этой сессии:**

1. **`CREATE OR REPLACE FUNCTION` не может изменить parameter list** — добавление optional p_calc привело к новой overload вместо replace. Решение: `DROP FUNCTION IF EXISTS public.<name>(BIGINT, TEXT)` перед `CREATE OR REPLACE` с новой сигнатурой. Транзакция корректно откатилась при первой попытке; добавил DROP statements и re-apply прошёл. **Канонический pattern для future migrations**: если меняешь сигнатуру (включая добавление optional defaults) — DROP first.

2. **`render_screen` визибилити evaluation = SQL fragment под EXECUTE** (L131-135): `'SELECT (' || cond || ')::boolean FROM public.users u WHERE u.telegram_id=$1'`. Это значит condition может вызывать functions — `has_active_safety_guards(u.telegram_id)` работает. Прецедент в проде — cancel-subscription с `can_cancel`.

3. **render_screen передаёт `text_key` в Python template engine as-is** (L189, L214) — placeholder substitution {count}, {family} в button labels НЕ работает. Substitution есть только для callback_data (L147-153). Architectural pattern для динамики button labels — отсутствует, требует engine extension (отложено в B3).

4. **business_data_rpc вызывается single-arg** (L83: `'SELECT public.' || rpc || '($1)'`) — все RPC с optional defaults поддерживают такой вызов.
