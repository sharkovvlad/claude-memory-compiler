## Sage VARIATION GUARD — wiring fix (PR #376)

**Контекст:** В handover `2026-06-08_sage-payload-meta.md` (PR #375) была открыта 🔴 задача task_7839624f: «дебаг, почему META `VARIATION GUARD` не появляется в `SAGE_MY_DAY_PROMPT` даже на 5-й my_day одного юзера со стампом в `ai_coach_logs`». В handover я выдвинул две гипотезы — SDK shape и фильтр `success=true`. Обе оказались мимо.

**Найденная причина (одна строка):** [services/sage.py:569](services/sage.py:569) — `regen_my_day_insight_cache` вызывал `generate_my_day_insight(ctx, day_summary)` **без** пробрасывания `rpc_caller`. У `generate_my_day_insight` default `rpc_caller=None` (был помечен `# reserved` ещё до PR #375). Внутри `_fetch_recent_sage_reactions` первой же строкой `if rpc_caller is None: return []` → пустой список реакций → `repeat_protein_stamp=False` → META не приклеивалась. food_log-путь не страдал — `handlers/food_log.py:139` всегда прокидывал `rpc_caller=_supabase.rpc`.

**Что отметено:**
- **SDK shape** — `supabase_client.SupabaseClient.rpc` возвращает `response.json()` прямо; для `RETURNS TABLE` это `list[dict]` нативно, без `APIResponse.data`. Никакого unwrap не нужно.
- **`success=true` фильтр** — на проде `psql -c "SELECT success, count(*) FROM ai_coach_logs WHERE context_type LIKE 'sage_%' AND created_at > now()-interval '2 hour' GROUP BY 1"` дал 19/19 строк с `success=t`. Фильтр в `mig 496` корректен.

**Почему unit-тесты не поймали:** `test_my_day_e2e_repeat_meta_flows_to_prompt` (PR #375, 38 тестов) инжектировал `rpc_caller` прямо в `generate_my_day_insight(..., rpc_caller=rpc)`. Тест доказывал «лист работает, если ему дать дверь», не «оркестратор пробрасывает дверь листу». Ровно паттерн `feedback_integration_tests_rpc.md` / KB [[concepts/sage-payload-meta-override-pattern]] — unit-моки скрывают wiring drift.

**Фикс:**
- [services/sage.py:569](services/sage.py:569) — `result = await generate_my_day_insight(ctx, day_summary, rpc_caller=rpc_caller)`.
- Удалён комментарий `# reserved` на параметре (он теперь активный).
- Новый тест `test_regen_my_day_cache_forwards_rpc_caller_for_repeat_meta` — мокает оба RPC (`get_day_summary` + `get_recent_sage_reactions`) и проверяет, что в финальный OpenAI-промпт попал `VARIATION GUARD`. Это тест на **wiring**, не на лист.
- Две стейл-ассерции в `test_sage.py` (count'ы RPC-вызовов в my_day cycle) обновлены: было 2→3 на success, 1→2 на LLM fail (теперь между ними всегда `get_recent_sage_reactions`).

### Что нашёл агент (sage fallback bug investigation)

**Инцидент 1 — u6378579500 07.06 13:12** — юзер был в edit-флоу (`status='editing_meal'`). [services/sage.py:180](services/sage.py:180) есть гейт `if (getattr(ctx, "status", "") or "") == "editing_meal": return None` — Sage **по дизайну** скипается. Этот гейт был anti-double-fire защитой эпохи n8n+Python (оба могли стрелять). После Stage 7b/7c (Edit Meal=Python единственный) гейт **отключает Sage на каждом редактировании** молча. Per-request ctx-cache (`dispatcher/context.py:178`) возвращает кэшированный `status='editing_meal'` даже после `clear_editing_state`, поэтому Sage стабильно молчит на edit'ах. **Юзер не видит Sage-строку, telemetry-строки в `ai_coach_logs` нет.**

**Инцидент 2 — u520145707 08.06 19:29 Борщ** — gpt-4o-mini превысил 5s таймаут (`sage_food_log_timeout_ms=5000` на проде, **не 1500** как я ранее предполагал; тот же юзер в 19:25 справился за 1935ms — обычный tail). Handler `wait_for` сработал на 5050ms → вызван `pick_food_log_fallback(ctx)` → **юзер УВИДЕЛ pre-baked ru-фразу** (одну из 5 вариантов). Но `pick_food_log_fallback` НЕ зовёт `_fire_sage_telemetry` → нет строки в `ai_coach_logs` → транскрипт показывает «отсутствует».

### 🎯 Большая системная находка — **telemetry hole**

В `services/sage.py` есть 9 путей возврата. **Только 1 (happy-path LLM)** пишет в `ai_coach_logs`. Остальные 8 (editing_meal skip, safety_guard skip, hard_skip namespace, no API key, openai err, json parse err, sanity reject, closed-budget leak override) + handler-уровневый `pick_food_log_fallback` (на outer timeout, `handlers/food_log.py:184`) — **невидимы**.

| Line | Path | Telemetry |
|---|---|---|
| 180 | editing_meal skip → None | ❌ |
| 186 | safety_guard skip → None | ❌ |
| 200 | hard_skip_namespace → pre_baked | ❌ |
| 206 | no API key → pre_baked | ❌ |
| 292 | openai err → pre_baked | ❌ |
| 304 | json parse err → pre_baked | ❌ |
| 323 | sanity reject → pre_baked | ❌ |
| 338 | closed-budget leak override → pre_baked | ❌ |
| 351 | LLM happy path | ✅ |
| `handlers/food_log.py:184` | outer timeout → pick_food_log_fallback | ❌ |

Это значит: **«2 пустые реакции за 2 дня» сильно занижает реальное число.** `/sage-tov` судья видит только happy-path реакции. Edit'ы (которых много) и любые timeout-fallback'и — за кадром. Без telemetry-фикса каждый будущий «пустой» инцидент — детектив в logs на 1-2 часа вместо `SELECT * FROM ai_coach_logs WHERE success=false`.

### Durable KB-уроки (для секции «Wiring gotcha» в [[sage-payload-meta-override-pattern]] и/или нового концепта)

1. **Telemetry coverage = contract.** Если у функции есть >1 return path, все они должны логироваться. Иначе observability — лотерея, и происходит то что произошло: ложная гипотеза «2 пустые реакции» при реальной «N невидимых fallback'ов» вытащила сессию в неправильном направлении.
2. **Anti-double-fire guards умирают вместе с эпохой.** Гейт `editing_meal → return None` имел смысл в n8n-эре. После Stage 7c (Python единственный source) — артефакт. Принцип: при удалении n8n-таргета пробежать grep'ом по `services/`/`handlers/` за gate'ами с упоминанием cutover'нутого пути.
3. **Per-request ctx-cache переживает RPC-мутации.** `dispatcher/context.py:178` кэширует ctx на запрос; RPC `clear_editing_state` обновляет БД, но не cache — следующий `ctx_loader` в том же handler-вызове вернёт stale `status='editing_meal'`. Это не баг, это контракт кэша; нужно знать.
