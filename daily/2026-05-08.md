## Сессия: 3 защиты против catastrophic semantic rollback (PR #32)

**Ветка:** `claude/cicd-pr-sanity-rollback-defense` (стэкована поверх PR #31)
**Worktree:** `agent-a86bfa94727b92eea`
**PR:** https://github.com/sharkovvlad/noms-bot/pull/32

### Контекст

Lesson 2026-05-08 (см. KB `concepts/release-protocol.md`): stale worktree + `git commit` → snapshot устаревших файлов других агентов попадает в commit → PR молча удаляет неделю работы. Спасло ручное `git diff origin/main..HEAD --stat`. Цель — автоматизировать.

### Что сделано

3 защиты, все стэк поверх инфры PR #31 (`.github/workflows/deploy.yml` + `hooks/pre-push.sh`):

1. **Защита 1 — `.github/workflows/pr-sanity.yml` + `.github/scripts/pr_sanity_audit.py`.** PR-level diff audit. Триггер `pull_request: [opened, synchronize, reopened]`. Чеки: total deletions >5000, удалённые миграции, удаление файлов без mention в title/body, per-file deletions >500 в файлах не из scope. При флаге → fail status + идемпотентный PR comment (обновляется, не плодится при каждом push). Пороги через repo variables `PR_SANITY_TOTAL_DELETIONS_LIMIT` / `PR_SANITY_PER_FILE_DELETIONS_LIMIT`.

2. **Защита 2 — расширение `pre-push.sh` (+82 строки, обратно-совместимо).** Перед каждым push в feature-ветку — `git rev-list --count $(git merge-base HEAD origin/main)..origin/main`. Пороги: >10 — warning, >50 — block (override `ALLOW_STALE_PUSH=1`). Если remote недоступен — тихо пропускается. Документировано в `HOOKS.md` (новая секция «Stale-worktree защита»).

3. **Защита 3 — `.github/workflows/nightly-pr-audit.yml` + `.github/scripts/nightly_pr_audit.py`.** Cron `0 3 * * *` UTC + `workflow_dispatch`. Через `gh pr list` обходит open PR, для каждого считает diff vs base. Suspicious → один аггрегированный Telegram-alert в admin chat (`TG_ADMIN_CHAT_ID=417002669`). Тишина при чистом state. Использует те же secrets что `deploy.yml`.

### Tested локально

- `python3 -c "import ast; ast.parse(...)"` — оба скрипта чисто.
- `yaml.safe_load(...)` — оба workflow чисто.
- `pr_sanity_audit.py` на синтетическом repo с 1 deleted migration → 2 чека срабатывают (critical + high), при `cleanup: remove` в PR title — high чек подавляется, critical migration-чек остаётся (by design — миграции всегда suspicious).
- `pr_sanity_audit.py` на 6000-line deletion → 3 чека срабатывают.
- `bash -n pre-push.sh` — синтаксис чист.

### Архитектурные решения

- **Стэкование PR:** база — `claude/cicd-github-actions-af8504` (PR #31), не `main`. Так PR #32 показывает только мои дельты. После merge #31 → автоматический rebase, diff схлопнется до scope только моих файлов.
- **Pin actions by SHA:** `actions/github-script@60a0d83039c74a4aee543508d2ffcb1c3799cdea` (v7.0.1, проверено через `gh api repos/actions/github-script/git/refs/tags/v7.0.1`). `actions/checkout` — тот же SHA что в `deploy.yml` (v4.3.1).
- **Идемпотентность PR comment:** маркер `<!-- pr-sanity-audit -->` в теле + поиск через `listComments` → `updateComment`. Не плодит comments при каждом push в PR.
- **Не блокирует merge:** на GitHub Free required-status-check для приватного репо не enforce. Только визуальный сигнал (красный check + comment). By design — соответствует 5-слойной процессной защите.
- **Telegram алерт rate-limit:** nightly шлёт максимум 1 message/сутки → не упрётся.
- **Defense-in-depth:** защита 1 ловит на push в PR; защита 2 — на push в feature-ветку (до создания PR); защита 3 — backup для случая если PR создан до того, как защита 1 появилась в main.

### Что НЕ сделано (out of scope, осознанно)

- Автооткат / закрытие suspicious PR (решение должен принимать человек).
- Migrations validation в pipeline (отдельная задача).
- `pr-sanity.yml` не использует ML / AI для классификации diff'ов (heuristics достаточны для текущего use case — catastrophic rollback всегда даёт 1000+ удалений).

### Lessons / gotchas

- **`git diff` в worktree:** `git rev-list --count` для stale-чека требует, чтобы `origin/main` был актуальным локально. Hook делает defensive `git fetch` если remote SHA не в локальном object DB — иначе чек был бы ненадёжен.
- **GitHub Free + required check'и:** для приватных репо без upgrade'а нельзя сделать required status check. Поэтому ставка на видимость (красный + comment + nightly Telegram), не на enforcement.
- **`actions/github-script` permissions:** workflow требует `pull-requests: write`. На приватном репо `GITHUB_TOKEN` имеет это по умолчанию для PR-trigger'ов — никаких extra secrets не нужно.

### Статус PR #32

Создан, target = `claude/cicd-github-actions-af8504` (PR #31). Не мержен — пользователю на review. После merge #31 надо ребейзить PR #32 на main; конфликтов не предвидится (мой scope непересекающийся с #31).
