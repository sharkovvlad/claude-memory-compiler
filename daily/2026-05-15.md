## mig 227 (renumbered from 217): calculate_user_targets v4 — PAL fusion + 'none' coefficients + target rebalance

PR [#75](https://github.com/sharkovvlad/noms-bot/pull/75) merged. Применена в проде через psycopg2 ещё до merge (под именем 217 — потом переименована из-за collision с параллельным агентом).

### Что починили (3 системные проблемы, найдены аудитом v3)

**1. PAL и `training_type` были decoupled.** Юзер с офисом (`activity_level=sedentary`, PAL=1.2) и тренировками 3×/неделю получал TDEE как чистый офисный работник — занижение на 15-25%. Самый массовый сегмент.

Fix: `app_constants.pal_training_bonus_{sedentary|light|moderate|heavy} = 0.15/0.10/0.05/0.00`. **Дифференциация по base activity_level** — защита от двойного учёта (heavy=1.725 уже включает intense training в Mifflin таблицах). `v_pal_adjusted = LEAST(pal_base + bonus, 1.8)`. Bonus применяется ТОЛЬКО если `training_type IN ('strength','cardio','mixed')`.

**2. Target_calories расходился с суммой БЖУ при срабатывании floor'ов** (fat_min_g_per_kg=0.8, carbs_min_g=50). Юзер видел на UI несогласованность.

Fix: Step 8 — conditional rebalance:
- `lose`: `v_target := LEAST(actual_kcal, ROUND(tdee))` — защита «нельзя похудеть на профиците».
- `gain`: `v_target := actual_kcal` безусловно — gain намеренно хочет target > TDEE.
- `maintain`: `v_target := actual_kcal` — floor-погрешность ±50 ккал OK.

**Lesson важный:** изначально написал безусловный `LEAST(actual, ROUND(tdee))` — это **сломало бы gain** (Test 2 sentinel: target должен быть 3090, при безусловном clamp получался бы 2808 = TDEE = maintain). Поймал на бумаге ДО apply, переписал в conditional. Документировано в KB [[concepts/personalized-macro-split]] (v4 section).

**3. `training_type='none'` (mig 119, для «реально не тренируюсь») попадал в `COALESCE(_, 'mixed')` fallback** → юзер получал 1.6 г/кг белка и 25% жира как mixed-tier. Это семантически противоречит выбору юзера.

Fix: `app_constants.protein_g_per_kg_none=1.2 / fat_pct_none=30` (= sedentary tier).

**Что НЕ трогали** (по решению ПМ — это политика, не баг): `cmd_select_training_skip → 'mixed'` mapping. NLM сначала предложил «починить» как баг, но grep по mig 119/190 показал — это сознательная политика тимлида (skip = «не хочу отвечать сейчас, дай дефолт» = mixed).

## Правило: «прод-first, не таблица-first» для digital twin

При работе над NOMS LiveOps Digital Twin (Google Sheets зеркало `calculate_user_targets`) договорились с владельцем продукта (вечер 15.05):

> **Если фичу всё равно нужно реализовать на проде — делать сразу в проде, а не в таблице. Таблица = зеркало prod, а не sandbox для proposals.**

**Почему:** двойной maintenance (новая логика в таблице → потом всё равно прод-migration → потом удаление proposal из таблицы) — лишняя работа без выгоды. Каждый proposal в таблице = долг, который придётся снимать. Sandbox оправдан **только** для Phase-X симуляций (где компонент будет реализован позже).

**Применение к `carbs_min_g_female=100` proposal:** удалили из таблицы. Моё научное мнение (зафиксировано в чате):
- Gender split для углеводов нужен только в edge cases (длинный дефицит >3 мес, low energy availability, симптомы)
- Для типичного NOMS-юзера (мягкий дефицит ≤20%) floor 50 г редко цепляется
- Когда цепляется — `floor_triggered_carbs=TRUE` сигнализирует юзеру что fast deficit ему не подходит → goal_speed уже решает проблему элегантнее
- Universal floor 50 г остаётся каноном

**Сопутствующее правило про хардкоды в таблице:**

> При любом UPDATE app_constants в проде, который меняет значение использованной в digital twin — обязательное ручное обновление таблицы в той же сессии. Иначе таблица расходится с prod и теряет статус «эталон».

Список отслеживаемых хардкодов в v6.0: `0,85`, `100`/`110` (obese offsets), `40` (obese floor), `1,2`/`1,375`/`1,55`/`1,725` (PAL), `1,8` (PAL cap), `2`/`1,4`/`1,6` (protein g/kg), `0,25`/`0,2`/`0,3` (fat %), `50` (carbs min), `0,9`/`0,85`/`0,8`/`1,08`/`1,1`/`1,15` (goal modifiers).

**Workflow для больших правок таблицы** (>3 ячеек):
1. Owner: Sheets → File → Download → CSV (активная вкладка)
2. Claude: правит через Read/Edit локально
3. Owner: Sheets → File → Import → Replace current sheet (форматирование сохраняется)

Минимум ручной работы owner'а — 2 клика. Лучше чем create_file MCP (теряет formatting).
