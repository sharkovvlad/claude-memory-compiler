### Что починили (3 системные проблемы, найдены аудитом v3)

**1. PAL и `training_type` были decoupled.** Юзер с офисом (`activity_level=sedentary`, PAL=1.2) и тренировками 3×/неделю получал TDEE как чистый офисный работник — занижение на 15-25%. Самый массовый сегмент.

Fix: `app_constants.pal_training_bonus_{sedentary|light|moderate|heavy} = 0.15/0.10/0.05/0.00`. **Дифференциация по base activity_level** — защита от двойного учёта (heavy=1.725 уже включает intense training в Mifflin таблицах). `v_pal_adjusted = LEAST(pal_base + bonus, 1.8)`. Bonus применяется ТОЛЬКО если `training_type IN ('strength','cardio','mixed')`.

**2. Target_calories расходился с суммой БЖУ при срабатывании floor'ов** (fat_min_g_per_kg=0.8, carbs_min_g=50). Юзер видел на UI несогласованность.

Fix: Step 8 — conditional rebalance:
- `lose`: `v_target := LEAST(actual_kcal, ROUND(tdee))` — защита «нельзя похудеть на профиците».
- `gain`: `v_target := actual_kcal` безусловно — gain намеренно хочет target > TDEE.
- `maintain`: `v_target := actual_kcal` — floor-погрешность ±50 ккал OK.

**Lesson важный:** изначально написал безусловный `LEAST(actual, ROUND(tdee))` — это **сломало бы gain** (Test 2 sentinel: target должен быть 3090, при безусловном clamp получался бы 2808 = TDEE = maintain). Поймал на бумаге ДО apply, переписал в conditional. Документировано в KB [[concepts/personalized-macro-split]] (v4 section).

**3. `training_type='none'` (mig 119, для «реально не тренируюсь») попадал в `COALESCE(_, 'mixed')` fallback** → юзер получал 1.6 г/кг белка и 25% жира как mixed-tier. Это семантически противоречит выбору юзера.

Fix: `app_constants.protein_g_per_kg_none=1.2 / fat_pct_none=30` (= sedentary tier).

**Что НЕ трогали** (по решению ПМ — это политика, не баг): `cmd_select_training_skip → 'mixed'` mapping. NLM сначала предложил «починить» как баг, но grep по mig 119/190 показал — это сознательная политика тимлида (skip = «не хочу отвечать сейчас, дай дефолт» = mixed).

## Migration collision guard — automated (pre-push hook + CI)

Триггер: mig 217 collision (этой ночью), плюс ≥3 предыдущих инцидента (mig 200 11.05 + два в апреле). Каждый раз — ручной renumber, потеря 30-60 мин, риск пропустить и получить два файла с одним NNN после merge.

Существовавшее правило в `CLAUDE.md` («перед push'ом проверить `ls migrations/ + gh pr list`») — ручное, не работало.

Сделано (PR ожидает create):
- **`.github/scripts/migration_collision_check.py`** — одна реализация для hook'а и CI. `--mode=local` читает пути с stdin, `--mode=ci` берёт через `gh pr view --json files`. Сверяет NNN с `git ls-tree origin/main migrations/` + `gh pr list --state open --json files`. На коллизию — печатает next free NNN + команду `git mv`.
- **`.github/hooks/pre-push.sh`** — новая секция «защита 4». Аккумулирует added `migrations/NNN_*.sql` из push-range (`--diff-filter=A`), после loop'а вызывает скрипт subprocess'ом. Override `ALLOW_MIG_COLLISION=1`.
- **`.github/workflows/pr-migration-collision.yml`** — CI backstop. Тот же скрипт, идемпотентный комментарий с маркером `<!-- pr-migration-collision -->`, fail status check.
- **`CLAUDE.md` секция «Миграции БД» п.8** — ручное правило заменено на ссылку на автоматику + override.
- **`.github/HOOKS.md`** — добавлена секция «Migration collision guard (защита 4)» + строка в таблицу hooks.
- **KB [[concepts/migration-collision-guard]]** — новый концепт, добавлен в index.md.
