## NLM sync rescue — noms-supabase-sync incident + preventive fixes

**Сессия:** подхвачена по handover'у предыдущего nlm-агента (см. daily/2026-05-19.md «Что осталось open»). Контекст: scheduled task `noms-supabase-sync` упал ночью на **70/70** загрузках с `Failed to get SOURCE_ID from registration response`. К утру 69 старых источников были удалены из блокнота `NOMS Supabase Data` (per pre-upload cleanup в SKILL.md), новые НЕ загрузились — блокнот без актуальных DB+RPC данных, осталось 116 (было 185).

**Что предыдущий агент НЕ доделал и в чём ошибся:**
1. Ноутбук не восстановил, скинул на следующего агента.
2. Утверждал «репо не git, либо нет remote — git log пустой». **Неверно:** `/Users/vladislav/Documents/NotebookLM+Claude/.git/` существует и в нём 80+ коммитов. Remote действительно нет — `git pull` бессмысленен (обновления через `pip install --upgrade notebooklm-py`), но репо — есть.
3. Утверждал «изменился upstream-эндпоинт Google NotebookLM, проблема в CLI». **Не подтвердилось:** тот же CLI 0.3.4 с теми же cookies через ~6 часов работал штатно. Прямой `notebooklm source add` вернул `Added source: 867e41b6-...`. Реальная причина — transient throttle/backend glitch на серии 69 delete + 69 add с малым delay=2.0.

**Root cause (с высокой уверенностью):** Google NotebookLM серверный RPC `o4cbdc` (ADD_SOURCE_FILE) при серии операций мог опускать source_id в payload → CLI 0.3.4 интерпретирует это как `Failed to get SOURCE_ID from registration response`. Это transient backend behavior, не поломка cookies/CLI/auth.

**Что сделано:**

1. **Recovery (primary fix):** Перезалил 69 файлов через `nlm.py batch-add-sources --delay 3.0`, исключив `users_2026-05-20.md` (загружен smoke-тестом ранее). Результат — **69/69 success, 0 errors.** Финальное состояние ноутбука: 180 sources, все `ready`.
2. **Upgrade `notebooklm-py` 0.3.4 → 0.4.1** в venv `NotebookLM+Claude/plugins/notebooklm-rag/.venv/`. В 0.4.x фиксы: «Per-domain cookie scoping on file upload» + «Server-state verification on timeout» (превентивно полезно при наших cookie-mixed warnings — `.google.es` duplicates). Smoke test после upgrade: auth ok, single upload ok.
3. **Bump delay в `noms-supabase-sync/SKILL.md`:** `--delay 2.0` → `--delay 3.0`. Файл живёт в `~/.claude/scheduled-tasks/noms-supabase-sync/SKILL.md` (вне git, change применится сразу).
4. **Retry/backoff в `nlm.py batch_add_sources`** (PR-style commit `4e91738` в локальном `NotebookLM+Claude` репо). 3 attempts с 5s/15s backoff, ловит маркеры: `SOURCE_ID`, `429`, `rate limit`, `timeout`, `500/502/503/504`, `temporarily`. Не-transient ошибки (file not found) fast-fail без ретрая. Happy-path и file-not-found code paths проверены.

**Defensive layers (3-level resilience):**

| Layer | Что делает | Trigger |
|---|---|---|
| L1: `--delay 3.0` | Reduce baseline throttle pressure | Always |
| L2: notebooklm-py 0.4.1 cookie scoping | Prevent mixed-domain cookie rejection | On upload |
| L3: retry с backoff | Recover from transient SOURCE_ID/429/5xx | On error |

Любая одна линия защиты могла бы предотвратить сегодняшний инцидент. Три вместе делают его маловероятным.
