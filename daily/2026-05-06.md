### Часть 1 — Бэкграунд: «Телепорт в Главное меню» (mig 178)

**Симптом:** на `onboarding_welcome` (status='new') клик "🌐 Язык" → выбор
"Українська" → бот выкидывал в Главное меню, language_code не сохранялся
(`'ru'` вместо `'uk'`). Следующий `/start` → "⚠️ Something went wrong".

**Root cause:** прод-версия `process_onboarding_input` (OID 161473) НЕ
соответствовала mig 173. Ветка `cmd_edit_lang` ставила
`status='changing_language'` вместо `push_nav('onboarding_welcome')`. Это
**третий случай stale-base regression** в проекте — после mig 167
(streak_freezes) и mig 042 (cron_check_streak_breaks). Скорее всего,
mig 175 (CREATE OR REPLACE через `pg_get_functiondef` снапшот ДО mig 173)
откатила mig 173.

**Цепочка бага:**
1. `cmd_edit_lang` → `process_onboarding_input` ставит `status='changing_language'`
2. `cmd_lang_uk` → router пускает в onboarding handler (status в
   ONBOARDING_STATUSES) → `dispatch_with_render` gate **не ловит
   `changing_language`** (gate: только `new`/`registration_step_%`/`restoring:choose`)
   → fallback в `process_user_input` (legacy)
3. process_user_input не знает `changing_language` для `cmd_lang_*` →
   generic save_via_callback с `clear_status=true`, `target_screen='settings'`
   → юзер видит settings/main_menu, язык не сохраняется
4. `/start` → snova не в gate → process_user_input возвращает error →
   `_envelope_from_rpc_result` → `_rerender_current_screen` → опять
   error → `_error_envelope` → "⚠️ Something went wrong"

**Migration 178 — что сделали:**
- ветка `status='new' + cmd_edit_lang/cmd_select_lang`: убрали
  `SET status='changing_language'`, добавили `push_nav('onboarding_welcome')`
- НОВАЯ ветка `status='new' + cmd_lang_*`:
  `set_user_language(callback)` ПЕРЕД render → peek+pop nav_stack →
  `render('onboarding_welcome')` (юзер сразу на новом языке, мы думали)
- НОВАЯ ветка `status='new' + cmd_back`: peek+pop nav_stack → top screen,
  fallback на welcome
- ВЕТКА 11 (`changing_language`) удалена — призрачный статус.
- Часть B: одной транзакцией с миграцией `UPDATE users
  SET status=COALESCE(previous_status, 'new'), previous_status=NULL
  WHERE status='changing_language'` (1 застрявший юзер).

**Verify mig 178:** 7/7 FSM walk pass через psycopg2, 122/122 pytest, p95
~47 ms (baseline 44 ms RTT — +3 ms на FSM логику).

- `process_onboarding_input` ветка `cmd_lang_*` синхронно обновлена
  на `(v_rpc_result->>'success')::boolean` (раньше было `'ok'`).

**Verify mig 179:** одна overload осталась (161473). FSM walk: cmd_lang_uk
→ language_code='uk', screen='onboarding_welcome', nav_stack=[] ✓.

### Часть 3 — Live QA выявил 3 параллельных бага (Tasks 3, 4A, 5)

После mig 178+179 владелец прогнал реальный сценарий через Telegram. Логи
0:13–0:16 MSK + наблюдения юзера:

#### Task 3 — Language Lag

**Симптом:** выбрал ES → следующий экран РЕНДЕРИТСЯ на RU → следующий
клик → на ES.

**Root cause:** `render_screen` (SQL) **не резолвит translations** —
эмитит только `text_key` + `language_code` в `telegram_ui`. Резолвинг
делает `services/template_engine.py:render_envelope`, читая
`ctx.translations` из **stale UserCtx, загруженного webhook'ом ДО RPC-вызова**.
После `set_user_language('es')` БД обновлена, но `ctx` всё ещё ru —
template_engine использует ru переводы для уже ES-юзера.

**Фикс (handlers/onboarding_v3.py `_envelope_from_rpc_result`):** после
получения rpc_result — если `telegram_ui.language_code` отличается от
`ctx.language_code`, перечитать ctx через `get_user_context(telegram_id)`
ПЕРЕД `render_envelope`. Стоимость: один view-direct GET (~44 ms RTT)
ТОЛЬКО при реальной смене языка. На штатных переходах — 0 RTT.

#### Task 5 — Silent Validation

**Симптом:** ввод 350 на шаге роста → бот молча перерисовывает запрос,
без сообщения об ошибке.

**Root cause:** SQL `set_user_height` fail возвращает
`render_screen('ask_height') || {validation_error:true, error_key:'invalid_height'}`.
В Python `_envelope_from_rpc_result`:

```python
raw_status = result.get("status")  # None — render_screen не возвращает status
if raw_status is None and isinstance(result.get("telegram_ui"), dict):
    status = "render"  # ← попадаем сюда
...
if status in ("validation_error", "error"): ...  # пропускаем — статус 'render'
# идём напрямую в render_envelope БЕЗ показа error_key
```

Флаг `validation_error` **тихо игнорировался**.

**Фикс:** после `envelope = render_envelope(...)` — если
`result.get('validation_error') is True`, аппенднуть `OutboundItem` с
локализованным error message. Defensive lookup translation:
1. `error_key` как есть (если уже dotted, типа `errors.validation_generic`)
2. `errors.<error_key>` (если ключ без точки — `invalid_height` →
   `errors.invalid_height`)
3. `errors.validation_generic` (final fallback)
4. `"⚠️ Try again."` (хардкод-самостраховка)

#### Task 4A — NUMERIC_RE расширение

**Симптом:** ввод `10000` на шаге веса → "Хм, не получилось распознать"
(легасный n8n RU-fallback).

**Root cause:** `dispatcher/router.py:NUMERIC_RE = ^\d{1,3}` —
позволял максимум 3 цифры. `10000` (5 цифр) НЕ match → router section 10
(`text_food_from_numeric_step`) → AI engine → fallback на хардкоженный
`'Хм, не получилось распознать. Попробуй ещё раз или напиши текстом.'`
в `n8n_workflows/03_AI_Engine.json:181`.

**Фикс:** `NUMERIC_RE = ^\d{1,5}([.,]\d{1,2})?$` — допуск 1-5 цифр.
После этого `10000` проходит в `set_user_weight` RPC, та возвращает
`error_key='invalid_weight'` (range 20-300) → юзер видит локализованную
ошибку через Task 5 фикс. AI engine больше не получает числовые
опечатки.

**Граница проверена тестом:** 6-значное число (`100000`) всё ещё → AI
fallback (рекордов веса/возраста/роста в 6+ цифр не существует).

#### Task 4B — НЕ делаем (решение тимлида)

Хардкод `'Хм, не получилось распознать'` в `n8n 03_AI_Engine` остаётся.
Риск трогать legacy AI engine (стабильно обрабатывает реальные food-логи)
выше, чем выигрыш — после Task 4A (NUMERIC_RE) AI engine больше не
получает числовые опечатки в онбординге. Hardcode зафиксирован как
тех. долг (см. ниже Gotcha #21).

### Дополнительные находки «с полей» (не блокеры)

**A. `editMessageText HTTP 400 "inline keyboard expected"` срабатывал 4 раза за 2 минуты QA.** Уже отрабатывается fallback'ом (mig 175 / Gotcha #17), но триггерится часто → каждый раз `delete+send_new` вместо `edit` → мерцание UI. Возможно усугубляет language lag (новое сообщение = новая отрисовка с stale ctx).

Mitigation: Task 3 фикс снимает language lag даже при таких fallback'ах. UI-мерцание остаётся как UX-долг.

**B. `template_engine: replace_existing without callback_message_id → fallback to delete_and_send_new`** — на text-input events (numeric_answer). Это намерение (text не имеет callback_query), но та же мерцающая отрисовка. Long-term — расследовать стратегию One Menu для text events.
