#### Task 3 — Language Lag

**Симптом:** выбрал ES → следующий экран РЕНДЕРИТСЯ на RU → следующий
клик → на ES.

**Root cause:** `render_screen` (SQL) **не резолвит translations** —
эмитит только `text_key` + `language_code` в `telegram_ui`. Резолвинг
делает `services/template_engine.py:render_envelope`, читая
`ctx.translations` из **stale UserCtx, загруженного webhook'ом ДО RPC-вызова**.
После `set_user_language('es')` БД обновлена, но `ctx` всё ещё ru —
template_engine использует ru переводы для уже ES-юзера.

**Фикс (handlers/onboarding_v3.py `_envelope_from_rpc_result`):** после
получения rpc_result — если `telegram_ui.language_code` отличается от
`ctx.language_code`, перечитать ctx через `get_user_context(telegram_id)`
ПЕРЕД `render_envelope`. Стоимость: один view-direct GET (~44 ms RTT)
ТОЛЬКО при реальной смене языка. На штатных переходах — 0 RTT.

#### Task 5 — Silent Validation

**Симптом:** ввод 350 на шаге роста → бот молча перерисовывает запрос,
без сообщения об ошибке.

**Root cause:** SQL `set_user_height` fail возвращает
`render_screen('ask_height') || {validation_error:true, error_key:'invalid_height'}`.
В Python `_envelope_from_rpc_result`:

```python
raw_status = result.get("status")  # None — render_screen не возвращает status
if raw_status is None and isinstance(result.get("telegram_ui"), dict):
    status = "render"  # ← попадаем сюда
...
if status in ("validation_error", "error"): ...  # пропускаем — статус 'render'
# идём напрямую в render_envelope БЕЗ показа error_key
```

Флаг `validation_error` **тихо игнорировался**.

**Фикс:** после `envelope = render_envelope(...)` — если
`result.get('validation_error') is True`, аппенднуть `OutboundItem` с
локализованным error message. Defensive lookup translation:
1. `error_key` как есть (если уже dotted, типа `errors.validation_generic`)
2. `errors.<error_key>` (если ключ без точки — `invalid_height` →
   `errors.invalid_height`)
3. `errors.validation_generic` (final fallback)
4. `"⚠️ Try again."` (хардкод-самостраховка)

#### Task 4A — NUMERIC_RE расширение

**Симптом:** ввод `10000` на шаге веса → "Хм, не получилось распознать"
(легасный n8n RU-fallback).

**Root cause:** `dispatcher/router.py:NUMERIC_RE = ^\d{1,3}` —
позволял максимум 3 цифры. `10000` (5 цифр) НЕ match → router section 10
(`text_food_from_numeric_step`) → AI engine → fallback на хардкоженный
`'Хм, не получилось распознать. Попробуй ещё раз или напиши текстом.'`
в `n8n_workflows/03_AI_Engine.json:181`.

**Фикс:** `NUMERIC_RE = ^\d{1,5}([.,]\d{1,2})?$` — допуск 1-5 цифр.
После этого `10000` проходит в `set_user_weight` RPC, та возвращает
`error_key='invalid_weight'` (range 20-300) → юзер видит локализованную
ошибку через Task 5 фикс. AI engine больше не получает числовые
опечатки.

**Граница проверена тестом:** 6-значное число (`100000`) всё ещё → AI
fallback (рекордов веса/возраста/роста в 6+ цифр не существует).

#### Task 4B — НЕ делаем (решение тимлида)

Хардкод `'Хм, не получилось распознать'` в `n8n 03_AI_Engine` остаётся.
Риск трогать legacy AI engine (стабильно обрабатывает реальные food-логи)
выше, чем выигрыш — после Task 4A (NUMERIC_RE) AI engine больше не
получает числовые опечатки в онбординге. Hardcode зафиксирован как
тех. долг (см. ниже Gotcha #21).

### Gotcha #23 (для KB) — Headless contract: language_code в `meta`, не в `telegram_ui`

`render_screen()` возвращает структуру:

`telegram_ui` — это **чистый UI** (Dumb Renderer контракт), без identity-полей. `meta` — контекст для Python handler'а. Любой Python код, который читает `telegram_ui.language_code` — баг.

### Gotcha #25 (для KB) — Conditional reply-kb re-attach узкий: forced attach для restart-like actions

`render_screen()` re-attach'ит reply-kb только при transition из text_input screen (mig 160 защита от `·` пузырей). Для **restart-like** действий (`/start`, потенциально cmd_back до root, smoke recovery) клавиатуру нужно прикреплять **принудительно** через `jsonb_set` на render result. Один `·` пузырь — приемлемая цена за надёжность.
