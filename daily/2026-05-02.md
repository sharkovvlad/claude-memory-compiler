## fix(indicator): миграция 165 — изоляция Python-стикера от One Menu UX (ветка `claude/fix-thinking-sticker`)

**Branch:** `claude/fix-thinking-sticker` (от текущего HEAD ветки `claude/ui-templates-fix`).
**Migration:** `migrations/165_isolate_indicator_message_id.sql` — applied на проде через psycopg2 + DATABASE_URL pooler.
**n8n PUT-ы:** `06_Indicator_Clear` (`jQn0nTxThFal4Kpe`, updatedAt 2026-05-02T08:16:48Z) + `03_AI_Engine` (`kjw4kkKMD0IqNALg`, updatedAt 2026-05-02T08:17:06Z).

### Проблема (расследование 02.05)

Тимлид попросил отключить узел `Send Thinking Sticker` в `03_AI_Engine` — юзеры видели **два thinking-стикера** на каждое фото/text/voice сообщение (один от Python proxy, второй от n8n). Расследование выявило **более глубокий архитектурный долг**:

- Python `maybe_send_indicator()` ([telegram_proxy.py:268-345](../../../telegram_proxy.py)) → `save_indicator_state` RPC (миграция 109, 21.04) → писал sticker `message_id` в `users.last_bot_message_id` с типом `'processing'`.
- Но `last_bot_message_id` — это **One Menu UX** ID последнего экрана-меню. Стикер «съедал» menu_id → следующий `delete_and_send_new` удалял стикер вместо предыдущего меню → старое меню оставалось висеть в чате.
- В `03_AI_Engine` 9 узлов отправки AI-ответа уже зовут `06_Indicator_Clear` → `clear_bot_message` → удаляли тот же `last_bot_message_id`. Это **скрывало** проблему частично, но не для всех путей и не закрывало архитектурно.

### Что сделано

**Миграция 165:**
- `ALTER TABLE users ADD COLUMN indicator_message_id BIGINT NULL` — изолированная колонка для стикера-индикатора.
- `CREATE FUNCTION clear_indicator_message(p_telegram_id) RETURNS TABLE(telegram_id, message_id)` — atomically read + nullify, returns old value для Telegram deleteMessage.
- `CREATE OR REPLACE FUNCTION save_indicator_state(...)` — теперь пишет в `indicator_message_id`, **НЕ трогает** `last_bot_message_id` и `last_bot_message_type`.
- Старый `clear_bot_message` НЕ ТРОГАЛИ — он по-прежнему работает с `last_bot_message_id` для One Menu UX path'ов.

**n8n `06_Indicator_Clear` (`jQn0nTxThFal4Kpe`):**
- Узел `DB: Get & Clear ID` query: `clear_bot_message(...)` → `clear_indicator_message(...)`. Возвращаемые поля те же (`telegram_id`, `message_id`), `Has ID to delete?` IF + `Telegram: Delete` остались валидны.

**n8n `03_AI_Engine` (`kjw4kkKMD0IqNALg`):**
- `Send Thinking Sticker` (telegram.sendSticker) → `disabled: true`.
- `Delete Thinking Sticker` (httpRequest deleteMessage по `Send Thinking Sticker.message_id`) → `disabled: true`.
- 9 веток `Clear Indicator (Success/Error/Edit)` → `06_Indicator_Clear` оставлены как есть. Теперь они зовут `clear_indicator_message` и удаляют именно Python-стикер из `users.indicator_message_id`.

### Verification (на проде сразу после apply)

- Колонка `indicator_message_id` существует, тип `bigint`, nullable.
- `save_indicator_state(417002669, 999999, 0, NULL)` → `indicator_message_id=999999`, `last_bot_message_id=3281` (не тронут — был меню), `last_bot_message_type='menu'` (не сменён на 'processing').
- `clear_indicator_message(417002669)` → returned `999999`, после: `indicator_message_id=NULL`. Идемпотентен на повторный вызов с NULL.
- 0 stale `last_bot_message_type='processing'` в БД до миграции — backfill не понадобился.
- noms-webhooks systemd alive (uptime 22h, get_indicator_context спокойно вызывается раз в минуту по продовому трафику).
- 06_Indicator_Clear PUT response: query содержит `clear_indicator_message`, не содержит `clear_bot_message` — подтверждено.
- 03_AI_Engine PUT response: оба узла `disabled=True`, все 49 нод сохранены (включая 3 `Clear Indicator (*)`).

### Архитектурные последствия

- **Дубль стикера:** устранён (n8n больше не шлёт второй стикер).
- **One Menu UX:** теперь корректен — стикер не вытесняет ID меню, `delete_and_send_new` найдёт правильный `last_bot_message_id` при следующей навигации.
- **Старое поведение в коде Python:** не менялось — Python вызывает RPC `save_indicator_state` по имени, тело RPC обновлено в БД. Никакого Python-кода и рестарта сервиса не потребовалось.
- **Race window** между apply миграции и PUT 06_Indicator_Clear был ~30 секунд. В это окно: Python писал в indicator_message_id (новое поведение), n8n зовёт clear_bot_message (читает last_bot_message_id). Эффект: 1-2 юзерских события могли потерять последнее меню (Clear бы удалил меню вместо стикера). На admin-аккаунте 417002669 в этот момент `last_bot_message_id=3281, type='menu'` — окно прошло без жертв.

### Что НЕ сделано (deferred)

- **Узкий путь `Send Step Reminder`** в 03_AI_Engine (httpRequest sendMessage в каком-то step-reminder flow) НЕ имеет downstream `Clear Indicator`. Если этот путь активен — Python-стикер там по-прежнему зависнет. Не блокирующее, отдельная задача.
- **Контракт 06_Indicator_Clear** теперь возвращает только 2 поля (`telegram_id`, `message_id`) вместо 3 (`message_type` убран). Если downstream callers ожидают `message_type` — сломаются. Проверка: ни один из 9 callers в 03_AI_Engine не использует return value (это executeWorkflow без обработки результата) → безопасно.
- **Старые stale записи** `last_bot_message_type='processing'` в БД — 0 шт., backfill не нужен.
