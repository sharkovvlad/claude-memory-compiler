### Полдень — prod incident #1: «бот молчит на Add food + AI»

Тех-лид сообщил два симптома:
1. «Добавить еду» (target=`menu`) — silent
2. AI распознавание — присылает 2 стикера и падает

Дебаг через SQLite n8n executions:
```
sqlite3 /home/noms/n8n/data/database.sqlite \
  "SELECT data FROM execution_data WHERE executionId IN (770, 771);"
```

**Найдено:**
- exec 770 (04_Menu) lastNodeExecuted=`Merge Data`, error=`Node 'Execute Workflow Trigger' hasn't been executed`
- exec 771 (03_AI_Engine) lastNodeExecuted=`Switch router`, error=`Node 'Execute Workflow Trigger 03' hasn't been executed`

**Root cause:** legacy sub-workflows используют n8n expression `$('NodeName').item.json.X` — runtime-ссылки на конкретные ноды по имени. При запуске через мой `Webhook B-1 Cutover → Adopt → ...`, нода `Execute Workflow Trigger` не выполняется в этой execution → ExpressionError → workflow падает.

**Phase 5 (flat payload, commit `e4444e0`) закрыл `$json` контракт, но НЕ закрыл `$('NodeName')` expression-ссылки** — это структурная проблема глубже Phase 5 scope.

### NLM consultation

Запросил NLM на cross-check архитектурных опций. NLM подтвердил scope (40+ нод суммарно ссылаются на trigger-ноды) и предложил **Опцию D**: переименовать мой Adopt → имя legacy trigger'а (например `Execute Workflow Trigger 03`), а старый trigger → `_Legacy`. Тогда downstream `$('Execute Workflow Trigger 03').item.json.X` найдут МОЙ adopt в истории execution.

**Audit Опции D показал критический gap у NLM:** при `admin_only=true` (наш staged rollout), не-админский трафик идёт через legacy 01_Dispatcher → executeWorkflow → внутри sub-workflow ищется trigger по типу (находит `_Legacy`, OK для запуска), но downstream-ноды ищут оригинальное имя (`Execute Workflow Trigger 03`), которого больше нет → **ломаются ВСЕ не-админы**. Не подходит для staged rollout.

Также n8n не позволяет два узла с одинаковыми именами (UI блок) — даже через PUT API сомнительно.

### Решение — Опция A (hotfix, 5 минут)

Сузить `TARGET_TO_PATH` до **только menu_v3** (единственный таргет где есть Python-обработчик через Phase 2). Все остальные таргеты → `forward.send` returns `unknown_target` → `_try_authoritative_path` returns False → fall-through на legacy n8n как до 30.04.

```python
# dispatcher/forward.py
TARGET_TO_PATH: dict[str, str] = {
    "menu_v3": "menu_v3",   # only Python-owned target
}
```

**Стратегический итог:** B-3 теперь работает по принципу «Python-prefix-matching» — обслуживает только то, для чего УЖЕ есть Python-обработчик. По мере того как новые sub-workflows будут переписываться в Python (следующий — onboarding), они добавляются обратно в `TARGET_TO_PATH`. Foundation сохранена.
