## Архитектура

`handle_menu_v3(update, ctx, decision, *, rpc_caller=None) → ResponseEnvelope`

Поток:
1. Из `RouteDecision` строится `p_payload` (callback_data / text / photo / voice / location), `p_cb_context` (is_inline + callback_query_id + callback_message_id), `p_action_type`.
2. ОДИН call в SQL bundle `dispatch_with_render` (migration 149).
3. Результат `telegram_ui` (та же форма что у `render_screen`) парсится через переиспользованный `services.render.parse_render_response`.
4. Если есть `callback_query_id` и в envelope нет `answer_callback_only` → prepend ACK FIRST (убрать спиннер до edit).

Status handling:
- `render` → telegram_ui → envelope (нормальный путь).
- `debounced` → empty envelope (либо single `answer_callback_only` для inline) + flags["debounced"]=True.
- `validation_error` → ACQ + send_new с локализованным `error_key` из ctx.translations.
- `error` → ACQ + send_new с `errors.generic`.
- `forward` → НЕ должен случаться для menu_v3, логируем и возвращаем empty.
- `render` без telegram_ui → error envelope "render_no_telegram_ui".

## Latency (бенчмарк на VPS Hetzner → Supabase EU pooler)

```
SELECT 1 baseline:        p50=42.4ms  p95=42.7ms

dispatch_with_render (skip_debounce=TRUE):
  cmd_get_stats:          p50=48.3ms  p95=265.8ms  (один cold-cache outlier)

  cmd_buy_freeze:         p50=46.0ms  p95=51.9ms
  cmd_back:               p50=47.3ms  p95=48.1ms
```

**Контекст:** текущий end-to-end через n8n ~1800ms. Чистый SQL ~46ms (44ms RTT + 2ms сам RPC). После переключения флага ожидаемая E2E latency: ~60-90ms (RTT + RPC + httpx + Python overhead) — т.е. **20-30× ускорение vs n8n**.

Все p95 далеко от gate 700ms. Watchlist: `cmd_get_stats` p95=266ms аномалия — возможно cold-cache, разово; повторить через неделю на больших объёмах.

## Phase 2 follow-up: 3 ADR fixes (commit 3810322)

После handover review request юзер прислал 3 ADR'а — каждый закрывает один из вопросов из секции 5.2/9 моего review-документа.

### ADR-1: Hot-reload feature flag в app_constants
- **Migration 156** (имя 155 уже занято другим агентом для cutover foundation — БД-эффекты не пересекаются, разные ключи в app_constants).
- INSERT `handler_menu_v3_use_python='false'` в `app_constants` (категория `feature_flags`).
- Триггер `trg_refresh_constants_cache` автоматически обновляет `app_constants_cache.data`.
- `v_user_context.constants` подтягивает кэш на каждом webhook'е → переключение флага через `UPDATE app_constants SET value='true'` без рестарта `noms-webhooks`.
- `webhook_server._menu_v3_flag_from_ctx(ctx)` читает из `ctx.constants["handler_menu_v3_use_python"]`.
- `.env HANDLER_MENU_V3_USE_PYTHON` остаётся kill-switch (cold-start, БД down).

### ADR-2: last_bot_message_id save (закрывает Section 5.2.b — потенциальный блокер)
- `services/envelope.py`: `SendResult.last_message_id: int | None` (additive поле).
- `services/telegram_send.py`: `_post` теперь возвращает `(ok, body)`, `_dispatch_item` — `(ok, new_message_id)`. Извлекается из `result.message_id` для `send_new` и `delete_and_send_new` (НЕ для edit/sticker/ACQ — у них либо нет нового message, либо не считается "меню" для One Menu UX).
- `webhook_server._send_and_persist(envelope, telegram_id)` обёртка: телеграм-send → если `last_message_id` заполнен → `save_bot_message` RPC (`p_message_type='menu'`, тот же контракт что узел "20. Save Bot Message" в n8n 04_Menu_v3).
- Без этого One Menu UX ломался — следующий menu jump удалял несуществующий message_id.

### ADR-3: Python pre-validation для numeric input
- `services/screen_validation.py`: lazy-loaded TTL-кеш (5 мин) загружает `ui_screens.validation_rules` + `workflow_states.state_code→screen_id`.
- `validate_text_input(status, text) → (is_valid, error_key)`.
- В `handlers/menu_v3.py`: для `action_type=='text'` валидация ДО RPC. Fail → error envelope с локализованным `error_key`, БЕЗ network roundtrip (~46ms экономии на каждом невалидном вводе).
- SQL-сторона (`validate_text_input` внутри `process_user_input`) ВСЕГДА перепроверяет — Python pre-validation не security-критично, чистая оптимизация и UX.
- Fail-open при ошибке загрузки кеша: пропускаем в SQL.
