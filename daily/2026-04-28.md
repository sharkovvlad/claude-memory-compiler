### Root cause

В `execution_data` SQLite свежих execs (40-48) нашлась реальная ошибка:

```
NodeOperationError: self-signed certificate in certificate chain
  at parsePostgresError (n8n-nodes-base/postgres)
```

Postgres-нода в `06_Indicator_Clear` (вызывает `clear_bot_message(417002669)` через credential `Postgres_pooler_6543`) падала на TLS handshake. Supabase pooler `aws-1-eu-west-1.pooler.supabase.com` использует chain, который **не входит в default CA bundle** Node.js внутри n8n контейнера.

В моём `scripts/n8n_migrate.py` (вчера) credential был создан с:
```python
"ssl": "require",
"allowUnauthorizedCerts": False,   # ← ошибка
```

На cloud n8n (откуда импортировались workflows) credential, видимо, был с `allowUnauthorizedCerts: true` (или другим ssl mode) — поэтому на cloud работало, на self-hosted упало.

### Каскад симптомов (один баг → 3 разных fingerprint)

Из 39 errors было 3 разных error message:
1. **Прямой:** `NodeOperationError: self-signed certificate in certificate chain` (постгрес-нода)
2. **Каскадный:** `NodeApiError: Bad request - please check your parameters` (Telegram API node после bad postgres data)
3. **Каскадный:** `failed to parse logic tree ((category.equal.processing,is_active.equal.true))` (PostgREST filter с битым input от preceding node)

Также в timeline вылезала более ранняя ошибка `Unauthorized: bad X-Noms-Secret header` (которая исчезла к моменту диагностики — возможно, transient env race при restart noms-webhooks). Эта ошибка отвлекла agent2 на анализ auth pipeline; реальная причина была глубже.

**Время до точного диагноза:** ~40 минут на анализ exec data в SQLite через несколько Python скриптов (т.к. public API `/api/v1/executions/{id}?includeData=true` не возвращает полный body, нужен прямой sqlite3 read).

### Fix

PATCH credential через internal `/rest/credentials/{id}` с full body:
```python
data = {
    "host": ..., "port": 5432, "database": "postgres", "user": ..., "password": ...,
    "ssl": "require",
    "allowUnauthorizedCerts": True,   # ← THE FIX
    "sshTunnel": False,
}
PATCH /rest/credentials/JW48yu2pfTUSntRn → HTTP 200
```

Hot-reload без рестарта контейнера. Workflow id ID не изменился → relink не требуется.

## Session: webhook_health alert-only mode (избавили от localhost trap)

**Контекст.** В 16:31 МСК пришёл админский алерт `🚨 NOMS webhook CRITICAL: Python proxy down AND unable to restore n8n webhook`. Расследование показало: network blip Hetzner на ~10-15 минут (`[Errno -3] Temporary failure in name resolution`). Все cron'ы (Reminder_meal_*, LeagueFomo, TonPaymentChecker и т.д.) тоже упали с DNS-ошибками одновременно — это сетевая проблема VPS, не наш код.

**Тихая ловушка (моя вина):** после big-bang switch'а (27.04 21:55 MSK) я обновил `N8N_WEBHOOK_URL=http://localhost:5678/webhook/...`. Но `crons/webhook_health.py` использовал ту же переменную в `_switch_webhook_to_n8n()` — то есть при срабатывании fallback он бы попытался `setWebhook(localhost)`, что Telegram отвергает (требует public HTTPS). До 28.04 fallback ни разу не срабатывал → ловушка незаметна. Первый network blip 28.04 16:23 МСК → fallback failed → critical alert.

**Решение (по совету пользователя — отказ от костылей):**
1. ❌ Не использовать cloud URL как fallback — split-brain риск (cloud workflows могут разойтись с self-hosted, пользователи получат непредсказуемое поведение).
2. ✅ **Pure alert-only mode:** cron больше не делает auto-setWebhook. Только Telegram alert админу. Если Python proxy упал — `systemd Restart=always RestartSec=5` поднимает сам. Если VPS network лежит — переключение webhook всё равно бесполезно.
3. ✅ **`FAILURE_THRESHOLD: 3 → 7`:** короткие network blips Hetzner (≤5 минут) больше не вызывают alert. Реальный простой ловится за 7 минут — Telegram буферизует updates 24h, нет user impact.

## Incident 28.04 17:27 MSK — VPS OOM из-за неработающего mem_limit + agent-параллелизма

**Симптом.** Бот не отвечает на reply-кнопки и текстовые сообщения. SSH к VPS висит / `Connection timed out during banner exchange`. ICMP ping работает (VPS жив на сетевом уровне), но TCP/8443 в таймаут.

**Last alert (от обновлённого webhook_health):** `🚨 NOMS webhook health FAIL` с detail `HTTP 503: {"status":"degraded","db":true,"n8n":false}`. Здесь видно ✅ работу нашего нового кода: формулировка про systemd hint, threshold 7, никакого split-brain fallback.

**Root cause (двойной):**

1. **Memory limit в compose НЕ работал.** Я ставил `deploy.resources.limits.memory: 1500M` (вчерашняя session при создании compose). Эта секция действует **только в Docker Swarm mode**. В обычном `docker compose up -d` (standalone) она **полностью игнорируется** — `docker inspect ... .HostConfig.Memory` возвращал 0 (no limit). Правильная нотация для standalone: `mem_limit: 1500m` напрямую под service.
2. **n8n memleak.** За 22 часа container вырос с ~250 MB → **2.7 GB anon-rss** (per OOM-killer log). Скорее всего:
   - Executions data накапливаются в memory cache (хотя есть EXECUTIONS_DATA_PRUNE=true, MAX_AGE=168ч)
   - Возможно бесконечные retry-loop'ы в каких-то нодах (вчера видели exec длительностью 201 сек)
   - Возможно n8n keep recent execution_data в памяти для UI display
3. **Параллельная работа агентов на prod VPS.** В пиковый момент у меня было 4-7 SSH-сессий одновременно. Один из агентов запускал `ripgrep` через всю кодовую базу — `rg` процесс ел 56% CPU. Combined с 72% RAM от n8n → load avg 22-46 при 2-CPU машине.

Cascade:
- VPS RAM 92 MB free / 0 swap → SQLite в n8n не может выделить память для transactions
- n8n возвращает HTTP 503 `Database is not ready!` на webhook calls
- Python proxy через `forward_to_n8n` получает 503 → /telegram/health возвращает 503 (n8n=false)
- webhook_health cron видит 7 fails (новый threshold) → admin alert
- VPS overloaded → SSH banner exchange timeout
- OOM killer наконец срабатывает в **20:52:28 UTC**, убивает `node` PID 1606111 (anon-rss 2.8 GB)
- systemd `noms-n8n.service` (`restart: unless-stopped` на container) автоматически поднимает n8n за ~10 секунд
- Recovery: load спадает, бот снова отвечает

### Fix

**Compose patched** через ssh-команду на VPS (`/home/noms/n8n/compose/docker-compose.yml`):

```diff
-    deploy:
-      resources:
-        limits:
-          memory: 1500M
-        reservations:
-          memory: 300M
+    mem_limit: 1500m
+    memswap_limit: 1500m
```

`docker compose down && up -d` → `docker inspect noms-n8n --format '{{.HostConfig.Memory}}'` теперь возвращает `1572864000` (1500 MB) — реально enforced. Если n8n опять разрастётся — Docker сам убьёт контейнер при достижении лимита, и systemd restart его за 5 сек, без OOM киллера всей системы.

**CLAUDE.md правило (новое):** агенты не должны делать долгие операции на prod VPS через SSH. Worktree на маке. Read-only короткие проверки (`curl`, `cat`, `journalctl --since`) — OK; долгий `grep -r`/`ripgrep`/`du -h /` — на маке после `scp`.

**Локальный compose файл отсутствует** — он только на VPS. Backup VPS-файла создан: `docker-compose.yml.bak.20260428_*`.

### Уроки

1. **`deploy.resources.limits` ≠ `mem_limit`.** В Docker Swarm работает первое, в standalone — второе. Compose v3+ syntax misleading; всегда проверять через `docker inspect ... .HostConfig.Memory` сразу после `up -d`.
2. **Agent-параллелизм на prod VPS = риск OOM.** На маленькой машине (3.7 GB RAM, 2 vCPU) даже один `ripgrep` агента в сочетании с другими процессами уводит load в 20+. Единственная защита — не пускать агентов на VPS сверх минимально необходимого.
3. **Новый webhook_health alert-only mode сработал правильно.** Threshold 7 минут не превышен ложно (network blips проигнорировал), но реальный 7-минутный outage поймал и отправил admin alert. Никаких setWebhook к cloud (split-brain risk избежан).
4. **OOM killer + systemd `restart: unless-stopped` = автоматический recovery.** Без human intervention за ~5 минут после OOM. Это hard safety net и она работает.
5. **n8n memleak — пока неизученный.** Долгосрочно: либо ограничивать `EXECUTIONS_DATA_MAX_AGE` агрессивнее (с 168h до 24h), либо разобраться с retry-loop'ами в нодах, либо принять как факт и просто полагаться на mem_limit + restart.

## Сессия 7 (2026-04-28 ~22:00 MSK): Plan-Б VPS stability — swap + n8n agressive pruning

**Контекст:** после OOM-инцидента сессии 6 — root cause фикс. Текущий стек: 281 MB n8n baseline → 2.7 GB anon-rss за 22ч → OOM kill. Утечка в Node.js heap, НЕ в SQLite (DB только 17 MB, 86 execution_entity записей всего за 26ч). Применяем защиту: host swap + Node heap cap + agressive pruning (защита от *будущего* роста execution_data).

### Что применено

**1. Host swap 2 GB:**
```bash
fallocate -l 2G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile
echo "/swapfile none swap sw 0 0" >> /etc/fstab
echo "vm.swappiness=10" > /etc/sysctl.d/99-noms.conf && sysctl -p /etc/sysctl.d/99-noms.conf
```
Verify: `swapon --show` → `/swapfile 2G`, `cat /proc/sys/vm/swappiness` → `10`.

Архитектурное решение (со-обсуждено): **swap для системы в целом, НЕ для контейнера n8n.** `memswap_limit: 1500m` в compose **оставлен равным `mem_limit`** — fail-fast: пусть Docker убивает контейнер при превышении 1.5 GB, чем спонсирует утечку host-swap'ом и получает "зомби-бота" с 15-секундной latency.

**2. n8n env vars (`/home/noms/n8n/compose/docker-compose.yml`):**
```yaml
- EXECUTIONS_DATA_PRUNE=true
- EXECUTIONS_DATA_MAX_AGE=48                    # было 168
- EXECUTIONS_DATA_PRUNE_MAX_COUNT=5000          # было 50000
- EXECUTIONS_DATA_SAVE_ON_SUCCESS=all           # KEEP — visibility для расследования 81% errors
- EXECUTIONS_DATA_SAVE_ON_ERROR=all
- EXECUTIONS_DATA_PRUNE_HARD_DELETE_BUFFER=24
- NODE_OPTIONS=--max-old-space-size=1024        # V8 heap cap 1 GB → агрессивный GC, видимая ошибка вместо kernel OOM
```
