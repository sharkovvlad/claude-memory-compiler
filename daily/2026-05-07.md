### Архитектурное решение

**Reply-keyboard прикрепляется один раз в правильный момент жизненного цикла** + точечные re-attach в edge cases.

3 уровня прикрепления (CASE в render_screen):
1. `v_is_reply_kb_screen=true` (input_type='reply_kb') — для status='new' welcome (existing)
2. `meta.reply_kb_entry=true AND previous_status ∈ text_input` — после ввода числа, toast «✅ Сохранено!» (mig 160 + mig 183 carrier_text_key)
3. `meta.attach_main_kb_unconditional=true` (mig 184) — onboarding_success_menu, post-completion

В `process_user_input.action='start'` — 30-day heuristic для long-absent юзеров (фикс mig 181).

### Gotchas (для KB) — добавлены

**Gotcha #26 (One-Time Attach Pattern):** Reply-keyboard для root reply_kb_entry screens НЕ через input_type='reply_kb' (это сборщик из ui_screen_buttons rows). Используй `input_type='inline_kb'` (или другой) + `meta.attach_main_kb_unconditional=true` чтобы render_screen вызвал `build_main_reply_keyboard()`.

**Gotcha #27 (Headless carrier text):** Hardcoded `·` carrier текст — антипаттерн. Используй chain: SQL emits `reply_keyboard.carrier_text_key` → Python резолвит из `ctx.translations` → fallback на `app_constants.icon_wave` → final `·`.

**Gotcha #28 (Multi-item envelope after complete_onboarding):** `_envelope_from_rpc_result` принимает `rpc_caller` для второго RPC. После `screen_id='onboarding_success'` — render второго экрана и extend items. Graceful degradation при exception.
