### Что НЕ переходило в пром (deprecated cleanup)

- Удалены `N8N_BASIC_AUTH_*` (deprecated с n8n 1.0+, заменены built-in user management — owner-аккаунт создаётся при первом visit `/setup`)
- Удалены `N8N_USER_FOLDER` и `DB_SQLITE_DATABASE` overrides (вызывали double-nested `/home/node/.n8n/.n8n` → EACCES)
- Удалён `N8N_RUNNERS_ENABLED` (deprecated в 2.x — runners enabled by default)
- Удалён `user: "1001:1001"` из compose (image имеет /home/node owned by UID 1000 mode 700; UID override ломает доступ)

### Ключевые tradeoffs

- **Изоляция через permissions, не через UID.** Файлы в `/home/n8n/data/` принадлежат UID 1000 (на хосте имя резолвится как `taskbot`), но `/home/n8n` mode 750 owner n8n:n8n — taskbot не имеет execute-bit, не может зайти в каталог. Чисто, не требует userns-remap.
- **SQLite, не Supabase Postgres.** Внутренние данные n8n (workflow definitions, executions, credentials, settings) — не бизнес-данные. SQLite читает workflow в ~0.5 мс vs ~30 мс через сеть к Supabase EU. Reasoning в плане секция 0.1.1.
- **5678 строго на 127.0.0.1.** UI доступен только через SSH-туннель. Никаких public endpoints без явного запроса.
