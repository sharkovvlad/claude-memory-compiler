### Session: Python Telegram proxy для typing-indicator (Dispatcher latency)

**Контекст:** юзер пожаловался на задержку до появления стикера "думающий NOMS". Замеры в n8n показывали 3.3s на reply-button и 6.2s на photo→AI. Корень: в 01_Dispatcher на критпути стояла цепочка `Quick Status Check` (Supabase, 100-300ms) → `Needs Indicator?` (Code) → `Send Indicator?` → `Send Early Indicator` (executeWorkflow 06_Indicator_Send, ~500ms). `Get User Context` ждал всю эту ветку из-за неправильных connections (оба output'а IF-ноды сходились в GUC).

**Попытка 1 (провалилась):** параллелизация connections в n8n — разорвать связь indicator-ветки с GUC. Применено (apply_fix.py), но сломало удаление стикера: 03_AI_Engine вызывает `06_Indicator_Clear` (читает `users.last_bot_message_id` через RPC `clear_bot_message()`), а из-за race condition `06_Indicator_Send` не успевал UPDATE'нуть БД до того как AI отвечал. Второй стикер висел в чате вечно.

**Попытка 2 (успех):** Python FastAPI proxy перед n8n.
- Telegram webhook переключен с `https://vlad1.app.n8n.cloud/webhook/<n8n_path>` на `https://89.167.86.20:8443/telegram/webhook` (self-signed cert upload через `setWebhook` + `certificate` file).
- Python отвечает 200 OK за **6-10ms**. В background: (1) fire-and-forget `maybe_send_indicator` → стикер/текст в Telegram + UPDATE `last_bot_message_id` через новый RPC `save_indicator_state`; (2) forward raw body в n8n для нормального routing'а.
- Сохранены все UX-инварианты: whitelist контента (menu-emoji / команды / флаги / callback_query), whitelist статусов (`registration_step_*`, `edit_weight` и др.), ротация 3 видео-стикеров + текст раз в день из `messages.analyzing` × 13 языков.
- Удаление стикера продолжает работать через существующий `06_Indicator_Clear` — Python синхронно пишет `last_bot_message_id` до того как AI может ответить (AI работает 3-7s, save — 100ms).

**Инфраструктура:**
- **HTTPS блокер** — VPS не имел SSL. Решение: self-signed cert в `/etc/ssl/noms/`, Telegram принимает при upload через setWebhook. uvicorn перезапущен с `--ssl-keyfile --ssl-certfile`.
- **Auto-fallback** — `WebhookHealthCron` в APScheduler каждую минуту дёргает `/telegram/health` (self-signed, verify=False). 3 fail подряд → `setWebhook` обратно на n8n URL + админ-алерт в чат `417002669`. Нет auto-flip-back (prevents flapping).

- **Параллельный агент опасен для webhook** — если другой агент делает деплой или rollback в той же сессии, он может переключить Telegram webhook. Нужна защита в `rollback_webhook.py`

**Архитектурные решения (для KB):**

- **Shared Screens contract для inline pickers:** 1 screen_id обслуживает онбординг + Profile edit. `workflow_states.state_code` → одинаковый `screen_id`, разные `next_step_code` в зависимости от FSM контекста. `ui_screen_buttons.visible_condition` = SQL expression (`u.status LIKE 'registration_step_%'`) для per-context вариативности (Skip button пример).
- **Callback convention = ground truth RPCs**, не specs mockups. `set_user_goal/activity/training_type/gender/notifications_mode` парсят `cmd_select_*`. `set_user_goal_speed` парсит `cmd_speed_*`. Specs mockups с `cmd_set_*` были устаревшие.
- **Picker save via callback** добавлен в process_user_input (Block H migration 110). SECURITY: whitelist `pg_proc WHERE proname LIKE 'set_user_%'` ДО `EXECUTE format('SELECT public.%I(...)')`. Pattern: `meta.save_via_callback=true + save_rpc + target_screen + clear_status`.
- **Checkmark prefix ≠ icon replace.** Когда button.is_current=true, render_screen НЕ меняет icon_const_key — просто ставит флаг. Dumb Renderer (frontend) префиксит `✅` перед существующим icon+label. Сохраняет визуальную семантику оригинальных иконок.
- **Top-level navigation fast-path:** cmd_get_profile/my_plan/settings/personal_metrics/help/delete_account всегда ведут на канонический screen_id независимо от nav_stack state. Сбрасывают nav_stack. Исключают "Profile → edit_speed" bug при накопленном stack.
