**Причина — структурная, не штамп.** PROACTIVE NAVIGATION в системном промпте называет навигацию «your core job». ### Race-fix food_log → my_day

Конкретный кейс: u6378579500 06-12 11:37 — food_log сказал «овощи», my_day тут же сказал «куриного филе» **через 0 секунд**. Оба пишут телеметрию через `asyncio.create_task`; my_day запрашивает `get_recent_sage_reactions` ДО того как food_log-строка попадает в `ai_coach_logs`. `VARIATION GUARD` видит пустой массив, штамп проходит.

**Фикс:** `handlers/food_log.py` передаёт `payload['noms_comment']` напрямую в `regen_my_day_insight_cache(just_emitted_food_log=...)`. Прокидывается до `generate_my_day_insight` который prepend'ит в `recent_reactions` ПЕРЕД `_detect_repeat_protein_stamp`. БД не трогается, race исчезает.

**3 wiring-теста** (того же класса, что bug 2026-06-08 rpc_caller forwarding):
- Leaf catches stamp from `just_emitted` (DB still empty)
- No-stamp input — guard не fires (asym guard)
- Orchestrator `regen_my_day_insight_cache` пробрасывает kwarg до leaf

**Гoча на пути:** `_PROTEIN_STAMP_RE` ловит «яйц» — мой первый тест-пример «Яйца и зелень — солидный заход» сработал как штамп. Поменял на «Ровный заход, всё на месте. 41-й день в серии 🔥» — без protein-stems. Durable урок: при создании тест-стрингов «должно НЕ срабатывать» сверять с регексом, а не интуицией.

## 23:00 — Bug fix: фото в profile-v5 substate пропадает (#387)

### Симптом (live tid=417002669, 22:24)

User в меню Профиль → ⚙ Настройки → Уведомления (`cmd_edit_notifications_mode` ставит `users.status='edit_notifications_mode'`) скинул фото еды. Бот молча проглотил — в чате остался прежний рендер «Настройки», `food_logs` пуст, в `journalctl`:

```
AUTHORITATIVE skip tid=417002669 target=ai reason=food_media — fall through to legacy
```

### Корень (5 минут расследования)

Stage 7 AI gate в `webhook_server._try_authoritative_path` ([webhook_server.py:1631](webhook_server.py)) принимал `food_media` только при `status in (registered, editing_meal, waiting_barcode_portion)`. `edit_notifications_mode` не в списке → skip → `forward_to_n8n` → **n8n `03_AI_Engine` DELETED 2026-06-08** → silent drop.

Router'у не за что винить — он корректно классифицировал `target=ai reason=food_media` (см. `dispatcher/router.py:1070`). Whitelist в webhook_server не обновлялся при добавлении profile-edit статусов — там до этого «ловить» было некому (n8n легасикомили AI).

### Класс gotcha

Тот же паттерн что [[concepts/fsm-state-whitelist-discipline]], но второй blast-сайт: НЕ роутерные frozensets, а **AI gate в webhook_server**. Когда legacy n8n fallback существовал — skip был benign (n8n обрабатывал). После `03_AI_Engine` DELETE 06-08 — skip превратился в silent drop. **Cutover N+1 уроков: каждый удалённый legacy-таргет требует grep по всем `fall through to legacy` log-сайтам в Python — что они теперь fallback'ят?**

### Фикс ([#387](https://github.com/sharkovvlad/noms-bot/pull/387), MERGED + LIVE)

1. `webhook_server.py`: import `PROFILE_V5_STATUSES` из `dispatcher.router`. Новый флаг `_is_profile_v5_food_escape` (`target=ai` + `reason=food_media` + `status ∈ PROFILE_V5_STATUSES`) расширяет гейт.
2. При срабатывании — fire-and-forget `set_user_status('registered')` чтобы picker не висел для NEXT запроса.
3. `text_food` из escape исключён: numeric input `88` в `edit_weight` мог бы перехватиться как «food». Защита, хотя router и так такой текст уводит на `menu_v3 → profile_v5_text_input` — defence in depth.
4. Новая лог-метка `AUTHORITATIVE_AI_PROFILE_ESCAPE` для телеметрии.
5. 2 unit-теста (`test_webhook_ai_engine_gate.py`): escape + reset, text_food NOT escapes. 13/13 в suite, 241 в смежных регрессионных suite зелёные.

### Для следующего агента

- Через 24-48ч: `journalctl | grep AUTHORITATIVE_AI_PROFILE_ESCAPE | head` — посмотреть как часто срабатывает. Низкая частота = редкий corner case, фикс достаточно. Высокая = надо думать о proactive reset status при выходе из picker'а (timeout? любая non-picker реакция?).
