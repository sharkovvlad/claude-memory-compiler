## Root Cause Analysis (важно для следующего агента)

### The Real Bug — n8n Cloud Switch v3.4 cache

**Доказательство (execution `9342` от 2026-04-18 11:44 UTC):**
- User нажал "Квесты"
- Menu Engine корректно выставил `action='rpc_fetch_render'`, `target_node='quests'`
- Action Router rules[1] — `rightValue='rpc_fetch_render'`, `outputKey='rpc_fetch_render'` (уникален)
- connections.main[1] → RPC Fetch Sub-Router
- **ФАКТИЧЕСКИ:** Action Router отправил items в **main[0]** (Render Sub-Router) — с `action='rpc_fetch_render'` пакетом

Это значит Switch **не проверяет rightValue** — просто льёт всё в `main[0]`. Проверено на нескольких executions (9324-9342), **100% попадание в `main[0]` независимо от action value**.

### Что мы пробовали (что НЕ делать снова)

| # | Попытка | Результат | Причина |
|---|---------|-----------|---------|
| 1 | Добавить Rule[25] phenotype_quiz в Menu Router | Misroute в Friends Info | Phase 2 start |
| 2 | Rule[25] пересоздать по шаблону Rule[24] | Misroute | Schema OK, но не помогло |
| 3 | Dedupe Rule[21] outputKey='back' → 'skip_meal' | Misroute | Параллельный агент откатил |
| 4 | IF bypass (Is Phenotype Quiz?) между IF и Menu Router | **Работало частично** | Quiz OK, 5 других routes broken |
| 5 | Dedupe Rule[21] + добавить Rule[27] в Menu Router + remove IF | Misroute | Runtime cache не сбросился |
| 6 | Code multi-output Menu Dispatcher (numberOutputs: 28) | **СЛОМАЛ PRODUCTION** | n8n validator reject `return [[items], ...]` |
| 7 | Action Router pattern (Menu Engine + 7-output Switch + 5 Sub-Routers) | **СЛОМАЛ PRODUCTION** | Switch ВСЕГДА кидает в main[0] |

**Уже откатывался через PUT 3 раза.** Rollback работает надёжно.

1. IF1: `menu_route === 'phenotype_quiz'`? TRUE → Build Quiz Screen + Push Nav (Quiz); FALSE → IF2

**Риск:** средний. IF ноды проверены в NOMS много раз и работают.
**Минусы:** некрасивая цепочка из 6 IF. Но это ВРЕМЕННОЕ решение.
**Время:** ~1.5 часа.

### Вариант B (настоящая победа над cache bug)
**Клонирование 04_Menu в новый workflow** + переключить executeWorkflow в Dispatcher:
1. POST /workflows → создать новый `04_Menu_v2` с исправлённым Menu Router
2. В `01_Dispatcher` → `Go to 04_Menu` executeWorkflow node — сменить workflow ID
3. Старый 04_Menu деактивировать (не удалять — rollback-safe)

**Риск:** средний. Новый workflow = новый runtime cache, без bug. Проблемы:
- executeWorkflow node в Dispatcher жёстко привязан к ID — надо менять
- Любые другие workflows ссылающиеся на 04_Menu (их не было, но надо проверить)

**Время:** ~2 часа.
