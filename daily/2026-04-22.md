## Session 11 — back_nav anchor fallback (migration 121)

**Agent:** Sonnet 4.6 (1M context), autonomous apply через psycopg2.

### Проблема (user report, 2026-04-22)

После кнопки с `meta.clear_status=true` (например, `cmd_lang_ru` на `edit_lang`) функция `set_user_language` срабатывает, `clear_status` меняет `users.status='registered'`, но в сочетании с top-level nav cleanup (migration 114) `nav_stack` может оказаться пустым. Результат: юзер визуально на экране (settings), жмёт «Назад» — `back_nav` возвращает `parent=NULL`, UI «виснет».

Migration 117 Block A уже добавила hierarchy-first приоритет в `process_user_input` (читает `ui_screens.back_screen_id_default` от `v_current_screen` до вызова `back_nav`). Но это работает **только если `v_current_screen` резолвится корректно**. После `clear_status`:
- `users.status='registered'` → `workflow_states['registered'].screen_id = NULL`
- `nav_stack=[]` → fallback `nav_stack->>-1 = NULL`
- итог `v_current_screen='profile_main'` (hard fallback) — НЕ настоящий visual screen → priority 1 ищет `back_screen_id_default` у `profile_main` (NULL) → priority 2 вызывает `back_nav(...)` без контекста → `parent=NULL`.

### Fix (migration 121)

**Файл:** `migrations/121_back_nav_anchor_fallback.sql`

**Изменения:**

1. **Новая сигнатура `back_nav`:** `back_nav(p_telegram_id bigint, p_current_screen text DEFAULT NULL) RETURNS jsonb`. Backward-compatible — однопараметровые вызовы продолжают работать.

2. **Anchor fallback логика:**
   - **Стек непустой:** обычный pop. *Edge-case:* если после pop стек стал пустым и parent=NULL, проверяется `ui_screens[popped].back_screen_id_default` — используется как parent.
   - **Стек пустой:** current screen определяется в порядке `p_current_screen` (hint от caller) → `workflow_states[users.status].screen_id`. Валидируется через `ui_screens`. Если найден — `back_screen_id_default` = parent.
   - **Всегда возвращается поле `source`** для debug: `stack | anchor_after_pop | anchor_from_param | anchor_from_status | empty_no_anchor | user_not_found`.

3. **`process_user_input` patched in-place:** две строки обновлены (priority-2 loop + cleanup pop) — `back_nav(p_telegram_id)` → `back_nav(p_telegram_id, v_current_screen)`. Идемпотентно через `DO $$ ... $$` block с проверкой `position()`.

### Verification — 7-scenario E2E

На ephemeral-user (telegram_id=99999999991), повторный rollback после dry-run + apply:

| # | Scenario | Input | Expected `parent` / `source` | Result |
|---|---|---|---|---|
| 1 | `stack=[]`, `p_current='settings'` | user hint | `profile_main` / `anchor_from_param` | ✓ |
| 2 | `stack=[]`, `status='edit_lang'`, no hint | status→workflow | `settings` / `anchor_from_status` | ✓ |
| 3 | `stack=[]`, `status='registered'`, no hint | (worst case) | `NULL` / `empty_no_anchor` | ✓ (caller handles hard fallback) |
| 4 | `stack=[profile_main,settings,edit_lang]` | normal pop | `settings` / `stack` | ✓ |
| 5 | `stack=[settings]` (pop → empty) | post-pop anchor | `profile_main` / `anchor_after_pop` | ✓ |
| 6 | nonexistent user | — | `NULL` / `user_not_found` | ✓ |
| 7 | `back_nav(telegram_id)` single-arg | backward compat | работает | ✓ |

**Post-commit independent verify:**
- `back_nav` signature: `(p_telegram_id bigint, p_current_screen text DEFAULT NULL)` ✓
- `process_user_input` содержит `back_nav(p_telegram_id, v_current_screen)`, старые паттерны отсутствуют ✓
- Реальный user 417002669 (`status=registered`, `nav_stack=[settings,edit_lang,settings]`): `back_nav(id, 'settings')` → `{parent:edit_lang, source:stack}` — работает корректно

### Покрытие

Anchor работает для **17 из 18 экранов** (все кроме `profile_main`, у которого `back_screen_id_default=NULL` — он и есть root). Это значит: после смены языка → settings → Назад корректно ведёт в `profile_main` через любой из 3 путей:
- nav_stack непуст → `settings` в стеке → pop → parent=`settings` или `profile_main`
- stack пуст → anchor_from_param (если caller передаёт `'settings'`) → `profile_main`
- stack пуст + status не дал hint → anchor_from_status через `status→edit_*→ui_screens[edit_*].back_screen_id_default='settings'`

### ⚠️ Остаточный кейс

Сценарий 3 (`stack=[]` + `status='registered'` + no `p_current_screen` hint) всё равно возвращает `parent=NULL`. Это ожидаемо — функция не может угадать visual screen без контекста. Caller `process_user_input` в этой ветке продолжает использовать hard fallback `'profile_main'` (priority 3). То есть UI всегда имеет валидный target, просто в этом edge case теряется hierarchy (но не ломается).

**Для полного решения** этого остаточного кейса потребовалось бы либо:
- (a) добавить `users.last_rendered_screen` колонку и обновлять её в render_screen
- (b) n8n явно передавать `current_screen` из callback payload в process_user_input

Оба — отдельная сессия. В рамках текущей задачи user фокус был на `back_nav` fallback — это сделано.
