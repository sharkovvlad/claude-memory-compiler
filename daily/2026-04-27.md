### Подводный камень №1: API key scope

Public API endpoint `POST /api/v1/workflows/{id}/activate` вернул HTTP 403 — мой API key не имел нужного scope (создавал с `workflow:create/read/update/delete/list/move`, но `workflow:execute` не было).

**Решение:** через session cookie на internal `/rest/workflows/{id}/activate` (full owner permissions без scope-checks). Cookie получается через POST `/rest/login` с `emailOrLdapLoginId` + `password`.

### Подводный камень №2: versionId required

`POST /rest/workflows/{id}/activate` вернул HTTP 400: `request.body.versionId required`. n8n 2.x ввёл workflow versioning — activation требует body `{versionId: "<current>"}` для optimistic concurrency.

**Решение:** для каждого воркфлоу — GET `/rest/workflows/{id}` → extract `versionId` → POST activate with body. Это нашло.

### Подводный камень №3: circular reference publish-time validation (главный)

После cookie-fix 9 из 13 воркфлоу активировались (HTTP 200). Оставшиеся 4 (01_Dispatcher, 02.1_Location, 02_Onboarding_v3, 04_Menu) и legacy 02_Onboarding + 04_Menu_v3 упали с HTTP 400:

```
Cannot publish workflow: Node "Go to X" references workflow Y which is not published.
```

n8n CE 2.x ввёл **publish-time validation**: при активации проверяется что **все referenced sub-workflows** в `executeWorkflow` нодах уже published. У NOMS архитектура **зацикленная** (02_Onboarding ↔ 02.1_Location ↔ 02_Onboarding_v3, 04_Menu → 02_Onboarding, 04_Menu_v3 → 02_Onboarding_v3 и т.д.) → ни один из узлов цикла не может стать первым published.

**На cloud этой проверки нет** (старая версия n8n) — поэтому cloud допускает active-with-references-to-inactive. На self-hosted (новая версия) — нет.

**Что испробовано (не работает):**
- PUT `/api/v1/workflows/{id}` с body `{active: true}` → HTTP 400 `active is read-only`
- env vars: нет видимых `N8N_*VALIDATION*` settings в env или /rest/settings
- 2 passes activation (sub-первые, parent-вторые) — циклы не дают разорваться
- Прямой UPDATE `workflow_entity SET active=1` в SQLite + restart → boot log: `Processed 19 draft workflows, 0 published workflows` — не регистрирует webhooks для drafts

**Что сработало (SQL workaround):**

n8n 2.x хранит published state в нескольких таблицах:
- `workflow_entity.activeVersionId` (NULL у drafts, = `versionId` у published)
- `workflow_published_version (workflowId, publishedVersionId)` — many-to-one mapping
- `workflow_history (versionId, workflowId, nodes, connections, ...)` — snapshot опубликованных версий

Forge published version для 6 drafts: для каждого
1. `INSERT OR IGNORE INTO workflow_history (versionId, workflowId, authors, nodes, connections, name, autosaved, description) SELECT ... FROM workflow_entity WHERE id=?`
2. `INSERT OR REPLACE INTO workflow_published_version (workflowId, publishedVersionId) SELECT id, versionId FROM workflow_entity WHERE id=?`
3. `UPDATE workflow_entity SET activeVersionId=versionId WHERE id=?`

Restart n8n — boot log: `Processed 19 draft workflows, 6 published workflows`. Все 15 показывают `Activated workflow ...`. POST /webhook/telegram-updates → HTTP 200 (раньше было 404).
