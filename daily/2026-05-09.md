**Контекст:** Сравнительный анализ вариантов ухода с n8n Cloud (self-host vs полная миграция на Python) и подготовка плана для следующего агента.

**Ключевые обсуждения:**
- Детальное сравнение варианта B (self-host n8n на VPS) vs варианта D (полный уход на Python) по критериям: latency, поддержка, стоимость
- Главный враг latency — cold-start n8n Cloud (1.3-1.5 сек), не исходящий HTTP к Telegram
- VPS Hetzner: 4 ГБ RAM (3 ГБ свободно), 2 CPU, 30 ГБ диска свободно, Docker не установлен
- На VPS живёт чужой проект `tasktracker-bot` — жёсткое ограничение: не трогать
- Реальный объём миграции на Python: ~240 нод в 14 воркфлоу, но 90% — тривиальные (httpRequest, code, postgres → 1-3 строки Python)
- Главный риск Python-миграции — `03_AI_Engine` с 5 langchain нодами (Vision, Transcribe), но `openai` SDK в Python решает

**Принятые решения:**
- **Стратегия двухфазная:** сначала self-host n8n (Шаг 0, убить cold-start за 1 день), потом постепенная миграция в Python (фазы 1-7)
- **n8n внутренняя БД — SQLite**, не Supabase (проще, n8n CE не поддерживает внешний Postgres официально для хранения воркфлоу)
- **Каждый n8n workflow → свой Python-модуль** (не один большой файл)
- **Критерий отмены n8n Cloud:** 3+ дня стабильной работы self-hosted
- **Route Classifier в изоляции переносить бессмысленно** — только вместе с целым потоком
- **Python Outbound Sender (Block B шаг 2)** — даст -30-60мс, не стоит усилий

**Выводы:**
- Self-host убирает cold-start полностью (n8n работает 24/7, не засыпает), latency Python↔n8n на одном сервере ~1мс
- Полная Python-миграция — цель на будущее, не экстренная мера; каждый этап даёт побочный выигрыш (меньше PUT-рисков, контракты в коде)
- Второй ИИ (НЛМ-консультант) дал полезный SWOT, но переоценил сложность миграции (3-6 недель) и не знал что n8n уже self-hosted с 27.04
