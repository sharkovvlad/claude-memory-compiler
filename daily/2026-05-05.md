### Блок A — get_user_context 404 (инцидент 12:48 MSK 5.05)

**Root cause:** `handlers/onboarding_v3.py:234` вызывал `get_user_context(telegram_id, rpc_caller=rpc_caller)`. Функция `get_user_context` в `dispatcher/context.py` имеет 2 code path: `query_caller` (view-direct GET к `v_user_context`) и `rpc_caller` (RPC). Передача `rpc_caller` форсировала путь через несуществующий RPC `get_user_context` → HTTP 404 → `ctx=None` → `ResponseEnvelope.empty()` → юзер видел тишину.

**Fix:** `handlers/onboarding_v3.py` — строка `return await get_user_context(telegram_id, rpc_caller=rpc_caller)` → `return await get_user_context(telegram_id)` (без аргументов → всегда view-direct GET).

**Примечание:** параллельно обнаружен второй баг — `_envelope_from_rpc_result` не обрабатывал `status=None` (render_screen возвращает результат без поля status). Добавлена эвристика: если `status=None` и есть `telegram_ui` → treat as render. Коммит `e9ffced`.

### Когда использовать meta.url

Любой случай, когда нужно открыть внешнюю ссылку (поддержка, документация, соцсеть, Web App) без callback на бот. Альтернатива (sub-screen с контактами через `meta.target_screen`) тоже валидна, но создаёт лишний клик. URL-кнопка — 1 клик, 0 RTT в Python/SQL.

**Watchlist для агентов:** при добавлении новых URL-кнопок:
- Только запись в `ui_screen_buttons.meta.url`. Никаких изменений Python/RPC.

- `meta` может одновременно содержать `target_screen` и `url` — но render_screen / template_engine применят `url` (он выигрывает в template_engine). Не смешивать.
