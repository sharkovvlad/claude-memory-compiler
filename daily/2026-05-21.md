### Лёссон 1 — visible_condition gotcha (надо в KB)

`render_screen` для `visible_condition` генерирует `SELECT (<cond>)::boolean FROM public.users u WHERE u.telegram_id = $1`. То есть условие подставляется как expression и читает ТОЛЬКО колонки `users`. Если condition = `can_cancel`, и колонки `can_cancel` в `users` нет — SQL падает с `UndefinedColumn`, который ловится EXCEPTION → кнопка молча скипается. Никаких логов user-side.

**Канонический pattern:** для условий на business_data → создать SQL function `<name>(p_telegram_id bigint) returns boolean STABLE` и вызывать в visible_condition как `public.<name>(u.telegram_id)`. Функция сама делает SELECT по нужным таблицам.

### Лёссон 2 — TON API v3 forward_payload shape change (надо в KB)

TON Center API v3 теперь отдаёт `comment: None` для большинства tx; memo живёт в `forward_payload` как сырая base64 BoC строка (не dict с `value` как было ранее).

**Code pattern:** `_extract_memo` static method с тремя fallback paths:
1. Top-level `comment` (legacy)
2. `forward_payload.get("value")` если `isinstance(fp, dict)` (legacy shape)
3. `base64.b64decode(fp)` + `re.search(rb"\x00\x00\x00\x00(\d{1,32})", raw)` для new shape (mig 300)

Для NOMS memo это ВСЕГДА цифры (telegram_id), поэтому digit-only regex безопасен и игнорирует трейлинг CRC bytes BoC.
