Код:
```js
const tzCity = md.timezone.split('/').pop().replace(/_/g, ' ');
const offsetStr = new Intl.DateTimeFormat('en-US', {
  timeZone: md.timezone, timeZoneName: 'shortOffset'

tzDisplay = tzCity + (offsetStr ? (', ' + offsetStr) : '');
```

Ограничение: город остаётся на латинице (Madrid), т.к. `Intl.DisplayNames` не поддерживает city names. Альтернатива в будущем — словарь TZ_CITY_NAMES × 13 langs или отдельная таблица `timezones_i18n` в БД.

### Bug 2: Language Back → Settings (не «Готово! Язык обновлён»)

**Контекст:** После Round 2 фиксов оставался deferred Bug 2 — при Профиль → Настройки → Язык → Back юзер видел "Готово! Язык обновлён" вместо возврата в Settings.

**Research findings (подтверждено через n8n API live state):**

- **Отдельного Language workflow НЕТ.** `gffHvoRJI2018qle` (02_Onboarding v1, inactive) содержит Return Path Router и `Parse Language`, но Dispatcher Route Classifier роутит `changing_language` status через executeWorkflow node `02_Onboarding` → **`Ksv1DhWKUIDtlhLy` (02_Onboarding_v3, active)**, а не v1.
- **v1 (gffHvoRJI2018qle) вызывается только ОДИН раз** — из 04_Menu `Go to Language` при первом нажатии "🌐 Язык" в Settings. v1 показывает picker (reply keyboard) и заканчивается.
- **Все последующие action (выбор флага, нажатие Back)** приходят в Dispatcher как text messages → роутятся в v3.
- **v3 → `Onboarding Engine` (Code)** — централизованный классификатор. Секция "3. status = changing_language": когда `isBack`, устанавливает `result.action = 'language_save'` + `result.is_back = true`. Но флаг `is_back` не используется downstream: Action Router матчит только по `action` → ветка `language_save` → `Save Language DB → Fetch Translations → After-Lang Builder → Send Telegram` (показывает "Готово!").

**Root cause:** Action Router не различает `language_save` vs `language_save+is_back`. After-Lang Builder всегда отправляет success-сообщение.

**Решение (минимальное):**

Изменён **один Code node** в v3 (Onboarding Engine):

**Было:**
```js
} else if (isBack) {
  result.action = 'language_save';
  result.language_code = d.language_code;
  result.return_status = returnStatus;
  result.is_back = true;
}
```

**Стало:**
```js
} else if (isBack) {
  result.action = 'status_and_send';
  result.status_update = { status: returnStatus, previous_status: null };
  // Build Settings screen payload (mirror 04_Menu Build Profile Sub-Screen / settings case)
  // ... (полный payload с текстом и inline_keyboard Settings)
}
```

`action = 'status_and_send'` использует **существующую** ветку Action Router[4] → `Update User Status → Send Telegram`. Никаких новых нод, никаких reroute connections.

**Изменения:**

1. **v3 `Merge Data`** — добавлены `country_code`, `timezone`, `last_bot_message_id` в return. Без них Build Settings в Engine не может отрендерить "Настройки: Страна/Часовой пояс".
2. **v3 `Onboarding Engine`** — isBack branch (236 chars → 2946 chars): `action = 'status_and_send'` + telegram_payload = Settings screen (inline keyboard идентичен `04_Menu Build Profile Sub-Screen settings case`).
3. **Никакие другие ноды/connections НЕ тронуты.** Action Router ветки 0-5 без изменений.

**Бэкап baseline:** `/tmp/onboarding_v3_backup_20260417.json`
**Rollback:** `python3 -c "import os, json, urllib.request; ..."` — PUT back from backup.

### Bug 2 Round 2: Settings показывал "Страна: —, Часовой пояс: UTC"

**Выявлено пользователем после E2E Round 1** (скриншот 11:40): Сценарий 1 работает — Back → Settings screen (без "Готово!"), НО поля `country_code` и `timezone` пустые. До Back было "Испания / Europe/Madrid (GMT+2)", после Back — "— / UTC, GMT+0".

**Root cause:** `Prepare for 02` в Dispatcher (typeVersion 3.2, `fields.values` mapping) передавал в v3 только 18 полей и **не включал `country_code`/`timezone`**. Merge Data v3 читала `input.country_code` → undefined → fallback `''`. Build Settings fallback → "—" / "UTC".

**Fix:** добавлены 2 поля в `Dispatcher → Prepare for 02 → fields.values`:
```js
{ name: 'country_code', type: 'stringValue', stringValue: "={{ $json.country_code || '' }}" }
{ name: 'timezone',     type: 'stringValue', stringValue: "={{ $json.timezone || 'UTC' }}" }
```
Итого fields: 18 → 20.

**Применено:** ✅ n8n API PUT Dispatcher (6XzdbQqpoUG0nsLe) + deactivate/activate

**Fix:** `JSON.stringify({...})` → `{...}` (plain object). n8n сам сериализует объект один раз.

|---|---|---|
| Get All Prices | `={{ JSON.stringify({p_telegram_id: $json.telegram_id}) }}` | `={{ {p_telegram_id: $json.telegram_id} }}` |
| Get Card Price | `JSON.stringify({p_telegram_id: $('Merge Data')...})` | `{p_telegram_id: $('Merge Data')...}` |
| Get Stars Price | same | same |
| Get Crypto Price | same | same |

### Phenotype Quiz — Phase 1 (migration 062, DB only)

**Контекст:** Экран «Complexión/Телосложение» в профиле уже рендерится (04_Menu → Build Ask Markup, case `edit_phenotype`), кнопка «✨ Hacer el test» шлёт `cmd_start_phenotype_quiz` — обработчика не было. Спек `.claude/specs/phenotype_quiz_spec.md` определяет 4 эвристических вопроса (визуальное распределение жира, тренировочный стаж, плотность тканей, история веса) и матрицу классификации в 4 фенотипа: monw / athlete / obese / default.

**План разделён на 2 фазы** из-за параллельного рефакторинга 02_Onboarding_v3 / 04_Menu (Picker Unification Strategy). Фаза 1 — автономная (БД/RPC/переводы). Фаза 2 (n8n quiz flow — 5 нод) отложена до завершения рефакторинга чтобы не конфликтовать со stale base.

**Research до написания SQL:**
- `calculate_user_targets(p_telegram_id BIGINT, p_save_to_db BOOLEAN DEFAULT false)` — получен полный код через temp n8n Postgres workflow. При `p_save_to_db=true` сама пишет `target_weight_kg`, `target_protein_g/fat_g/carbs_g`. `COALESCE(v_user.phenotype, 'default')` — NULL-safe. LBM Proxy: `monw → weight × phenotype_monw_modifier`, `obese → height - phenotype_obese_offset_male/female` с MIN 40, `default/athlete → weight_kg`.
- 30 юзеров в БД, все с `phenotype='default'` — backfill не нужен.
- `target_weight_kg` заполнен только у 1 юзера (тестовый 417002669 = 99.0), у всех остальных NULL — заполняется при первом вызове `calculate_user_targets` с `p_save_to_db=true`.
- Все 13 языков уже имеют `profile.phenotype_hint / start_test / body_type / current / phenotype_monw / phenotype_athlete / phenotype_obese / phenotype_standard` — intro-экран рендерится корректно, новые intro-ключи не нужны.
- Callback «Hacer el test» = `cmd_start_phenotype_quiz` (подтверждено в live 04_Menu → Build Ask Markup).
- 01_Dispatcher Route Classifier: `isMenuCallback = callback.includes('cmd_') && !callback.includes('cmd_select_')` — `cmd_phenotype_*` уже ловится через catch-all `cmd_`. Менять диспетчер не нужно.

**Migration 062 — применено:**

1. **RPC `classify_phenotype(p_telegram_id, p_q1, p_q2, p_q3, p_q4) → JSONB`:**
   - Классификационная матрица (obese проверяется ПЕРВЫМ — тот же q1/q2/q3 паттерн что monw, но BMI > 27 распределяет):
     ```sql
     WHEN q1='c' AND q2 IN ('a','b') AND q3='a' AND BMI > 27 THEN 'obese'
     WHEN q1='c' AND q2 IN ('a','b') AND q3='a'             THEN 'monw'
     WHEN q1='a' AND q2='c'          AND q3='c'             THEN 'athlete'
     ELSE 'default'
     ```
   - Сохраняет `phenotype` + `phenotype_answers` (JSONB).
   - Сбрасывает `status='edit_phenotype' → 'registered'` (safe: CASE пропускает иные статусы).
   - Вызывает `calculate_user_targets(id, true)` — единая точка правды для пересчёта макро.
   - Возвращает `{success, phenotype, bmi, answers, targets}`.

   - `q1-q4_prompt/a/b/c` (16)
   - `result_title`, `result_recalculated` (2)
   - `result_explanation_monw/athlete/obese/default` (4) — Sassy Sage-персонализация под каждый фенотип
   - HTML parse_mode (`<b>` в prompts), апострофы удвоены для SQL

**Важное наблюдение про CTE в PostgreSQL 12+:** первоначальная верификация использовала CTE + подзапрос к users в одном `SELECT jsonb_build_object(...)`.
