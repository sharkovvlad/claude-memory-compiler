### P0 #1 — Stars Subscriptions broken: PROVIDER_ACCOUNT_INVALID

**Symptom:** owner tested Stage 2 monthly Stars purchase from 786301802 +
417002669. Telegram payment modal opened with
`Error: PROVIDER_ACCOUNT_INVALID` (no other text).

**Investigation:**
* Bot logs: `createInvoiceLink → HTTP 200 OK`, valid `t.me/$...` URL
  returned. NOMS Python side fine.
* The error fires CLIENT-SIDE when Telegram opens the URL — MTProto
  payment backend rejects subscription invoice.
* Identical payload minus `subscription_period` (i.e. quarterly/yearly
  one-time Stars) works fine — confirmed via `sendInvoice` flow.
* Telegram docs don't mention any BotFather prerequisite for
  `subscription_period`. WebSearch + WebFetch on changelog + payments-stars
  + bug tracker — no hits.

**Conclusion:** `subscription_period` requires an undocumented BotFather
configuration step for the bot owner (likely
`/mybots → @nomsaibot → Bot Settings → Stars Subscriptions`, OR a
whitelist request to `@BotSupport`).

**Hotfix:** `_STARS_SUBSCRIPTION_PLANS = frozenset()` in
`handlers/payment.py`. All Stars plans (monthly + quarterly + yearly)
route through one-time `sendInvoice` (pre-Stage-2 behavior). Mig 365
infrastructure stays live in DB (idempotent, dormant). Cancel/resume
Python branches stay (inert — no recurring subs exist).

**KB:** `concepts/stars-subscriptions-botfather-prereq.md` (HUB).
Re-enable checklist + live-test protocol + `provider_token` omit rule
(must NOT be empty-string — must be ABSENT entirely).

### P0 #2 — Stacked-PR base-change gotcha

**Symptom:** owner merged #213 → #214 → #215 in sequence via GitHub UI.
All three showed MERGED in `gh pr list`. BUT `origin/main` was missing
mig 365 and Stage 3 Python. Migration-collision CI warning persisted
on stale pages.

**Cause:** to clear the CI red on collision check, agent ran
`gh api PATCH base=<feature-branch>` on #214 and #215. When owner clicked
«Merge pull request», GitHub merged into the **feature branch base** —
not into `main`. No warning. PR shows MERGED truthfully (it WAS merged,
just not where expected).

**Recovery:** opened PR #216 from `claude/stars-recurring-stage2-creation`
(which now contained all stage 2+3 commits via the chain of intermediate
merges) directly to `main`. Owner merged → main complete.

**KB:** `concepts/stacked-pr-base-change-gotcha.md` (HUB) —
3 safer workflows (sequential merge with «Update branch», single
combined PR, or scripted API-merge with manual re-targeting before each
merge). Recovery recipe documented.

### Lessons reusable in code

* `provider_token` for `currency='XTR'` must be **omitted entirely**, not
  empty-string. Documented in
  `concepts/stars-subscriptions-botfather-prereq.md`.
* Multi-stage migration features: prefer **single combined PR** unless
  intermediate stages provide standalone user value. Don't
  `gh api PATCH base=` to silence collision-CI — instead use sequential
  merge with «Update branch» button (1 click per PR).
* When recovering a botched stacked merge: find the last stage's branch
  (it accumulates all prior stage merges), open recovery PR to main, done.
