### Session 7 — Headless RPC hotfix (migrations 095 + 096)

**Контекст:** Session 6 закрыла data bugs (091-093) в Profile v5 pilot, но core routing bug оставался: `process_user_input` падал с SQLSTATE 55000 "record v_button is not assigned yet" при любом inline click на экране НЕ `profile_main`. Execution 9589 подтвердил deterministic reproduce. Три independent agent (Session 6 explore + NLM consult + Session 7 deep diagnose) сошлись на том что bug не в n8n, а в RPC — hybrid state resolution между `users.status` и `nav_stack` не работает.

**Архитектурное решение (NLM-validated hybrid):**

```sql
-- Step 1: workflow_states (process states: edit_weight/age/height, registration_*, payout_*)
SELECT ws.screen_id, ws.save_rpc, ws.next_step_code
  INTO v_current_screen, v_save_rpc, v_next_status
  FROM workflow_states WHERE state_code = v_user.status;

-- Step 2 (новое): idle state → top of nav_stack (JSONB negative indexing)
IF v_current_screen IS NULL THEN
    v_current_screen := v_user.nav_stack->>-1;
END IF;

-- Step 2.5 (defensive): existence guard (legacy 04_Menu пушит 'profile' который НЕ в ui_screens)
IF v_current_screen IS NOT NULL
   AND NOT EXISTS (SELECT 1 FROM ui_screens WHERE screen_id = v_current_screen) THEN
    v_current_screen := NULL;
END IF;

-- Step 3: deaf fallback
IF v_current_screen IS NULL THEN v_current_screen := 'profile_main'; END IF;
```

**Migrations applied (autonomous via psycopg2):**

1. **`095_hotfix_routing_and_records.sql`** — 5 изменений в `process_user_input`:
   - Hybrid cascade (workflow_states → nav_stack->>-1 → profile_main)
   - `SELECT INTO ... LIMIT 1 + IF FOUND` вместо FOR-LOOP с `v_button := NULL` (избавились от 55000 gotcha)
   - `push_nav(v_next_screen)` вместо `push_nav(v_current_screen)` + `IS DISTINCT FROM` guard (stack top = current screen)
   - Status reset на `cmd_back` расширен: `input_type='text_input' OR v_save_rpc IS NOT NULL`
   - DDL `CHECK (length(callback_data) <= 64)` на `ui_screen_buttons`
   - Pre-check DO-block для валидации существующих rows перед ADD CONSTRAINT

2. **`096_rpc_validate_screen_exists.sql`** — defensive patch поверх 095:
   - Existence guard (Step 2.5) между nav_stack и fallback — нужен потому что legacy 04_Menu (вызывается из Dispatcher при reply-click "Профиль") пушит `'profile'` в nav_stack, а этот screen_id не существует в `ui_screens` (только `profile_main` после 086 rename)
   - Повторный data cleanup nav_stack (migration 092 была single-shot)
   - Validation back_nav result: если `v_back_result.parent` не в ui_screens → fallback

### Session 8 — UX паттерн text save + n8n PUT v3 (migrations 097 + 098 + 2 PUTs)

**Контекст:** Session 7 закрыла core routing bug (inline кнопки работают), но после text input save пользователь не видел confirmation. Session 8 фиксит этот UX gap и закрепляет паттерн для всех будущих text_input flows.

**Решение (NLM-validated + user refinement):** UX-паттерн "👌 reaction на user message + delete prompt + render обновлённого экрана". Legacy-style toast "✅ Вес обновлён" отклонён в пользу Headless-пуристского подхода (чистый чат, инструктивная реакция, никаких лишних сообщений).

**Migrations applied (autonomous via psycopg2):**

1. **`097_rpc_override_strategy_for_text.sql`** — override `telegram_ui.render_strategy='delete_and_send_new'` когда `p_action_type='text'` OR `callback_message_id` пуст. Починка root cause BUG 1: RPC возвращал `replace_existing` + пустой callback_message_id → editMessageText(message_id=0) → Telegram 400.

2. **`098_rpc_success_reaction.sql`** — эмитит `telegram_ui.success_reaction='👌'` в RPC когда `p_action_type='text' AND v_save_ok`. Контракт для n8n: workflow читает поле и ставит реакцию через `setMessageReaction`.

**n8n PUTs в 04_Menu_v3 (ID `ju0h4WStPZX54EfR`, 2 round'а после approval):**

**Round 1 PUT — 4 changes (15 → 18 nodes):**
- **Fix Switch Render Strategy:** `mode="rules"` (v3.4 cache bug — `delete_and_send_new` ошибочно роутился в output 0 / editMessageText) → `mode="expression"` с явным int-mapping `{replace_existing:0, delete_and_send_new:1, send_new:2}[render_strategy] ?? 2`.
- **Early Answer Callback Query:** переместил из `RPC → ACQ` в `Extract Payload → ACQ` параллельно. Убирает "query too old" errors (ACQ теперь < 100ms после click, не 1-2 сек после RPC render).
- **Set Message Reaction node** (parallel branch): `Dumb Renderer → Is Success Reaction? → setMessageReaction` — ставит 👌 на `$('Execute Workflow Trigger').item.json.message.message_id`.
- **Save Bot Message node** (fire-and-forget): `HTTP sendMessage → save_bot_message RPC` — поддерживает актуальный `last_bot_message_id` для One Menu UX.

**Round 2 PUT — 3 bugfixes после first live test (18 → 19 nodes):**

После первого live smoke test обнаружилось 3 бага в round 1:
- **sendMessage clobber:** после `HTTP deleteMessage` нода `HTTP sendMessage` читала `$json.text`, но `$json` = ответ deleteMessage API (`{result: true}`), не данные Dumb Renderer. Fix: `$('Dumb Renderer').item.json.text` (bypass clobber, CLAUDE.md rule 1).
- **success_reaction не форвардился:** Dumb Renderer JS не включал `success_reaction` в final return object. Fix: добавлен `success_reaction: ui.success_reaction || ''` в output.
- **Нет typing indicator:** UX — юзер не видит что бот обрабатывает. Fix: добавлена нода `HTTP sendChatAction` (action='typing') параллельно от Extract Payload, fire-and-forget до RPC.

**Верификация:**
- **SQL:** text save `'98'` возвращает `render_strategy='delete_and_send_new'` + `success_reaction='👌'`. Callback с cb_ctx возвращает `replace_existing` без reaction (правильно). Callback без cb_ctx — defensive `delete_and_send_new`.
- **Live smoke test user:** юзер ввёл "88" → увидел typing indicator → 👌 на своём сообщении → старый prompt удалился → появился новый экран "Мои Данные: Вес 88 кг". **End-to-end работает.**

**Паттерн закрепился в системе для ВСЕХ будущих text_input flows:**
- Не нужно менять n8n для каждого нового save RPC (`set_user_age`, `set_user_height`, будущие edit flows)

### Session 9 — Profile Agent Dossier Pure Headless (migrations 100 + 101 + 2 n8n PUTs)

**Цель:** перевести `profile.main_text` из legacy "Simple Profile" в формат **Agent Dossier** (Досье Агента) с сохранением Pure Headless архитектуры. Запланировано в Session 8 roadmap как migration 100.

**Ключевой архитектурный момент (critique-driven):** первая попытка (external AI → `100_..._v2_full.sql`) была **отклонена** из-за hardcoded эмодзи в templates (`📂 🕵️ 📅 💰`), Markdown `**bold**`, monolithic text × 13 языков. Два ревьюера (NLM + другой AI) указали на anti-pattern. User подтвердил: идём **Option A — Pure Headless / TMA-ready**.

**Разделение слоёв (single source of truth):**
- Данные → RPC (raw JSON, zero text)
- Переводы → `ui_translations` (атомарные ключи, существующие `profile.title_agent`, `profile.sage_*`)
- Иконки → `app_constants` (эмодзи)
- Layout → `profile.main_text` (один universal шаблон на 13 языков)
- Склейка → n8n Dumb Renderer JS (multi-pass interpolation)

**Phase 1: Расширение Dumb Renderer JS (04_Menu_v3 PUT, регрессия passed)**

Добавлена поддержка нового template syntax:
- `{tr:dotted.path}` — lookup в `ui_translations[lang_code].content.dotted.path`
- Nested braces: `{{goal_{goal_type}}}` → сначала `{goal_type}`→"lose" → потом `{{goal_lose}}`→📉
- Nested в `{tr:}`: `{tr:profile.sage_{sage_tier}_{sage_quote_index}}` → `{tr:profile.sage_premium_2}` → quote
- Multi-pass с iter<5 (stability cap), backward compatible для existing `{var}` / `{{const}}`

После PUT — regression test всех Profile v5 экранов (my_plan, personal_metrics, settings, help, delete_account) — все работают как раньше.
