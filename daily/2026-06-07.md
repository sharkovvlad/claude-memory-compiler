## Sage closed-budget guard hardening — PR #352 (services/sage.py, без миграции)

**Запрос owner'а:** разобрать переписку с внешним ИИ про «жиры зашкаливают / динамический пересчёт лимитов жир↔углеводы» + почему дешёвая mini советует есть при превышении калорий.

**Решения по существу (durable):**
- Динамический пересчёт макро-целей внутри дня — **ОТКЛОНЁН** (подтверждено owner'ом). Калорийный потолок уже балансирует жир↔углеводы; двигать сохранённые цели — плохой UX + over-engineering (аудит уже отклонял). «Гибкая диета» реализуется как **coaching-сообщение навигатора**, а не как сдвиг чисел.
- Внешний ИИ переврал пример 1 (75%, белок отстаёт → совет постного белка) — это НЕ баг, это правильный навигатор при открытом бюджете.
- Белок vs калории: на конкретный день при закрытом бюджете **недобор белка = меньшее зло** (оба кумулятивны; добор белка едой на профиците саботирует цель). Белок добирается ВЫБОРОМ еды внутри бюджета (рано в дне), не добавкой сверху. На длинном горизонте белок критичен (мышцы) — но защищается изо дня в день, не перееданием в один вечер.

**Корень бага (114%-день «как насчёт куриного филе»):** мы сами слали модели противоречивый сигнал — `_macro_remaining_line` печатал «P 27g remaining» при калориях +361 over. Модель логично предложила добрать белок. Гард «не звать есть при закрытом бюджете» жил только как закопанная негативная инструкция → слабые модели её роняют (LLM negation + curse-of-instructions: малые модели <42% при множестве конфликтующих правил; GPT-4o/Claude держат).

**4 детерминированных усиления (Python+промпты, без миграции):**
1. `_macro_remaining_line` при `kcal%≥100` → `budget CLOSED (over by N kcal)` вместо per-макро remaining.
2. `_budget_directive` — одна позитивная директива в КОНЕЦ промпта (модели весят конец), без enum-decode и без «do not». Оба пути (food_log + my_day).
3. Commit-then-generate JSON: модель декларирует `budget`/`next_meal_suggested` ДО `text`.
4. Детерминированный гейт (язык-агностичный, БЕЗ регекса): реально закрыт + модель `budget=open` ИЛИ `next_meal_suggested=true` → откат (food_log→pre-baked fallback, my_day→static SQL insight). Регекс по 13 langs сознательно НЕ делали (anti-pattern KB).

**Durable lesson:** когда критичный гард зависит от слабой LLM — (а) не шли противоречивый payload (per-макро remaining при закрытом бюджете); (б) вердикт давай ОДНОЙ позитивной директивой в конце, вычисленной в Python, не закопанной негацией; (в) commit-then-generate (поле-решение до текста) дёшево повышает соблюдение; (г) safety-net через структурное self-report поле, не через 13-язычный регекс.

### CI-фикс mig 481: JSONB shallow-merge guard

PR #353 упал на `pr-jsonb-merge-guard` — гард грепает токен `\bcontent\s*\|\|`. Мой генератор писал `SET content = content || jsonb_build_object(...)` (top-level shallow-merge, тот самый паттерн P0 26.05 «282 ключа стёрты»).

**Важно:** мой apply БЫЛ безопасен (внутренний `COALESCE(content->ns,'{}') || новые_leaves` сохранял соседей — проверено: 0 потерь, все langs ~1004 ключа). Но полагаться на suppress не стал.

**Фикс:** переписал генератор на guard-safe alt #2 — per (lang, namespace): `jsonb_set(content, '{ns}', COALESCE(content->'ns','{}') || jsonb_build_object(...), true)`. Нет токена `content ||` → гард чист. Идемпотентно переприменил (DB уже была корректна). **Gotcha:** литерал `content ||` в КОММЕНТЕ SQL тоже флагается — переформулировал коммент.

**🔑 Durable:** при bulk-update `ui_translations` НИКОГДА `content = content || jsonb_build_object(ns, ...)` (top-level shallow-merge стирает siblings ns). Только `jsonb_set(content,'{ns}', COALESCE(old)||new, true)` per namespace. И не писать литерал `content ||` даже в комментах миграции. KB [[jsonb-shallow-merge-antipattern]].
